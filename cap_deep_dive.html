The Core Idea

BIG PICTURE

Most robot systems need to answer a question like this: "Put the red block in the empty bowl." How does the robot turn natural language into motor commands? There are fundamentally different approaches:

ApproachHow it worksCore problem
Symbolic Parser Language → semantic parse → planner → robot actions Brittle on new phrasing or open-ended tasks
End-to-end Neural Text + image → big neural policy → actions Requires large amounts of robot demonstration data
Code as Policies ★ Language → LLM writes Python → code calls perception + control APIs → robot acts Expressiveness limited by available API surface
The fundamental reframe: Instead of teaching a robot to understand English instructions, you teach it to write Python. The LLM generates executable logic — not a narrative of steps — that runs directly on the robot hardware.
Python generated_by_llm.py
# user: "Put the red block a bit to the right of the blue bowl." # generated by LLM — not hand-written: target_pos = get_pos("blue bowl") + np.array([0.1, 0.0]) put_first_on_second("red block", target_pos) # → [ACTION] move red block → [0.5, 0.2]

Foundations: LMPs vs. CaP

DEFINITIONS

The paper distinguishes two related concepts. A Language Model Program (LMP) is any program generated by a language model and executed on a system. Code as Policies (CaP) is a robot-specific LMP that reads perception outputs and calls control APIs.

Python lmp_vs_cap.py
# LMP = any program an LM generates and executes # CaP = robot-specific LMP # Example CaP: "stack the blocks in the empty bowl." empty_bowl_name = parse_obj('empty bowl') block_names = parse_obj('blocks') obj_names = [empty_bowl_name] + block_names stack_objs_in_order(obj_names=obj_names) # The helper is generated too (hierarchical gen): def stack_objs_in_order(obj_names): for i in range(len(obj_names) - 1): put_first_on_second(obj_names[i + 1], obj_names[i])

Why Code Is the Right Interface

DESIGN RATIONALE

Code provides structural affordances that plain English plans lack. Every one of these is used directly in the paper's real-robot experiments:

Sequences

robot.pick(obj) robot.place(obj, target)

Conditionals — Reactive Behavior

if detect_object("orange"): robot.set_velocity(x=-0.1)

Loops — Feedback Policies

while not detect_object("apple"): robot.set_velocity(y=0.1)

Arithmetic + Geometry

target_pos = get_pos("red block") \ + [0.1, 0]
Third-party libraries: The paper explicitly encourages importing NumPy and Shapely for geometric reasoning. A natural-language plan might say "move a bit right." Code says target = get_pos("block") + np.array([0.05, 0.0]). Ambiguity is grounded into computation.
Pythonnumpy_geometry.py

SAFE_GLOBALS = { "get_pos": get_pos, "put_first_on_second": put_first_on_second, "robot": robot, "np": np, } def safe_exec(code_string): banned = ["import ", "__", "exec(", "eval("] for token in banned: if token in code_string: raise ValueError(f"Unsafe token: {token}") local_scope = {} exec(code_string, SAFE_GLOBALS, local_scope) return local_scope # retrieve ret_val etc.

Prompt Design — From the Official Repo

OFFICIAL PROMPTS

The actual prompts from code-as-policies.github.io/prompts/ reveal the full strategy. Two prompt types are composed at runtime: a high-level UI prompt and a function generation prompt.

Multi-prompt composition: Different LMP instances handle different roles — parse_obj_name, parse_position, parse_question, and fgen each use their own prompt. The top-level generated code calls these as functions with natural language arguments, triggering further LLM calls at runtime.
🗂 High-Level UI Prompt code-as-policies.github.io/prompts/tabletop_ui.txt

Key design insight from the actual prompts: parse_position('a 30cm vertical line on the right with N points') is itself a call to another LMP instance — one that returns a list of 2D coordinates. The top-level code orchestrates a multi-LLM pipeline through normal Python function calls with natural language strings as arguments.

Hierarchical Code Generation

KEY CONTRIBUTION

The paper's most important technical contribution. Instead of one monolithic code block, the LLM calls undefined helpers — then the fgen prompt is invoked to define them recursively.

❌ Flat Code Generation

# "stack the blocks in the empty bowl." empty_bowl = find_empty_bowl() blocks = get_blocks() for block in blocks: pick(block) place(block, empty_bowl) empty_bowl = block

One monolithic chunk. LLMs struggle here; harder to debug.

★ Hierarchical Code Generation

# "stack the blocks in the empty bowl." empty_bowl = find_empty_bowl() blocks = get_blocks() stack_objects([empty_bowl] + blocks) # ↑ undefined → fgen prompt fills it in: def stack_objects(obj_names): for i in range(len(obj_names) - 1): put_first_on_second( obj_names[i+1], obj_names[i])

Decomposed, readable, each piece semantically named.

Pythonnonmatching_colors.pyfrom real experiments
# "Put the blocks in bowls with non-matching colors." blocks = get_obj_names_by_type('block') bowls = get_obj_names_by_type('bowl') pairs = get_nonmatching_color_pairs(blocks, bowls) # ← fgen defines this for block_name, bowl_name in pairs: put_first_on_second(block_name, bowl_name) # fgen then produces: def get_nonmatching_color_pairs(blocks, bowls): pairs = [] for block in blocks: for bowl in bowls: if get_color(block) != get_color(bowl): pairs.append((block, bowl)) break return pairs
Think like a programmer solving a hard problem: write the main logic calling named helpers, then fill in the helpers. Code LLMs benefit from the same decomposition strategy, and the paper empirically confirms it improves both RoboCodeGen and HumanEval scores.

Closed-Loop Reactive Policies

BEYOND PLANNING

A critical distinction: CaP is not just a high-level skill sequencer. Generated code can sit directly in the perception-action feedback loop, making it a genuine low-level policy.

Pythonclosed_loop_policy.py
# "While the red block is to the left of the blue bowl, # move it to the right 5cm at a time." while get_pos('red block')[0] < get_pos('blue bowl')[0]: target_pos = get_pos('red block') + [0.05, 0] put_first_on_second('red block', target_pos) # Feedback loop: # observe position of red block # ↓ # still left of blue bowl? # ├── yes → move 5cm right → loop # └── no → stop
Pythonbehavioral_commonsense.py
# "Do it again but faster, to the left, and with a banana." # LLM assigns concrete numeric values to vague language: while not detect_object("banana"): robot.set_velocity(x=0, y=-0.2, z=0) # "faster" → 0.2 (from LLM's learned prior) # "to the left" → negative y direction
Paper's explicit claim: CaP can represent reactive low-level policies like PD or impedance controllers — not just waypoint-based policies. This is qualitatively different from earlier LLM-for-robotics work (e.g. SayCan) that stopped at skill sequencing.

Pythoncap_full_example.py
import numpy as np # ── World state ──────────────────────────────────────────────── world = { "red block": {"pos": np.array([0.1, 0.2]), "type": "block"}, "blue bowl": {"pos": np.array([0.4, 0.2]), "type": "bowl"}, } # ── Perception APIs (backed by MDETR/ViLD on real robot) ─────── def get_obj_names(): return list(world.keys()) def get_pos(name): return world[name]["pos"] def detect_object(name): return name in world # ── Control APIs ─────────────────────────────────────────────── def put_first_on_second(obj_name, target): target_pos = get_pos(target) if isinstance(target, str) \ else np.array(target, dtype=float) world[obj_name]["pos"] = target_pos print(f"[ACTION] move {obj_name} → {target_pos}") class Robot: def set_velocity(self, x=0.0, y=0.0, z=0.0): print(f"[ACTION] velocity=({x}, {y}, {z})") robot = Robot() # ── LLM-generated code ───────────────────────────────────────── # Instruction: "Move the red block a bit to the right of the blue bowl." generated_code = ''' target_pos = get_pos("blue bowl") + np.array([0.1, 0.0]) put_first_on_second("red block", target_pos) ''' # ── Safe execution ───────────────────────────────────────────── SAFE_GLOBALS = { "np": np, "get_obj_names": get_obj_names, "get_pos": get_pos, "detect_object": detect_object, "put_first_on_second": put_first_on_second, "robot": robot, } for banned in ["import ", "__", "exec(", "eval("]: if banned in generated_code: raise ValueError(f"Unsafe token: {banned}") local_scope = {} exec(generated_code, SAFE_GLOBALS, local_scope) print("Final red block position:", world["red block"]["pos"]) # → [0.5, 0.2]

Strengths, Limits & Deepest Intuition

SYNTHESIS

What This Paper Gets Right

  • Uses the LLM where it's strongest — comment→code completion aligns with pretraining
  • Makes reasoning executable — ambiguity resolved at code level, not left to interpretation
  • Low data requirements — prompting, not end-to-end retraining on robot demos
  • Hierarchical decomposition is empirically validated — benefit transfers to general HumanEval
  • Fully interpretable — inspect generated code to see exactly what the robot "thinks"
  • Multi-LMP composition — specialized LLM instances for parsing, geometry, and QA

What CaP Is NOT

  • A neural architecture trained end-to-end on robot sensor streams
  • A diffusion policy or flow-matching approach
  • A VLA predicting action tokens directly (like RT-2)
  • A multimodal transformer taking image embeddings (like PaLM-E)
  • Able to "code its way out" of missing perception or control primitives

Limitations (from the paper)

Perception Ceiling

Bounded by what perception APIs can describe. Can't reason about undetectable things.

Control Ceiling

Bounded by available control primitives. No primitive → no action.

Prompt Saturation

Prompts can become too long if too many primitive parameters are exposed.

Feasibility Assumption

Cannot know in advance whether generated code will physically succeed.

Should an LLM talk to a robot in English? Not necessarily.
Sometimes the best language for robot reasoning is Python.

For tasks involving coordinates, loops, perception checks, thresholds, or geometric reasoning — code is a strictly better intermediate representation than English. CaP reframes robot instruction-following as program synthesis for embodied control.
