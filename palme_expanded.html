§01 The Grounding Problem: Why LLMs Are Blind Robots

Large Language Models (LLMs) like PaLM, GPT-4, and their predecessors are remarkable reasoning engines. They can write essays, solve math problems, produce code, and carry on nuanced conversation. But there is a fundamental limitation when you try to deploy them in the physical world: they have no perceptual connection to reality.

Consider the instruction: "Bring me the rice chips from the top drawer." An LLM can parse the syntax, understand the intent, and even know that rice chips are a snack. But it cannot answer the questions that actually matter for execution: Which drawer is the top one? Is it open or closed? Where exactly in the drawer is the rice chips bag? Is there a red bag or a blue bag? Without answers to these questions — answers that require real-time visual perception — the instruction is unexecutable.

⚡ Core Insight: The Grounding Problem

Grounding refers to the connection between linguistic symbols and their referents in the physical world. "Red block" is just a string of characters until the agent can see which block is red and where it is in 3D space. Prior LLM-based robot systems attempted to bridge this gap through a lossy intermediate representation — natural language descriptions of what vision systems see. PaLM-E proposes eliminating this bottleneck entirely by injecting raw continuous sensor data directly into the LLM's embedding space.

Why the Caption-Then-Reason Approach Fails

The obvious workaround is to run a separate vision model, convert its output to text ("I see a red block on the left side of the table, approximately 20cm from the edge"), and feed that caption into the LLM. This approach has several fundamental problems:

Information Loss

Converting a rich visual scene to text inevitably discards information. Exact spatial coordinates, fine-grained textures, occlusion patterns, and relative object geometry are all compressed into imprecise language. "The block is near the edge" doesn't tell you if it's 2cm or 15cm away.

No Closed-Loop Reasoning

If the vision captioner makes a mistake, the LLM has no way to know — it sees only the caption. The LLM can't look back at the raw image to double-check an ambiguous description. Errors in the vision pipeline propagate unchecked to the reasoning layer.

Serial Bottleneck

The pipeline is strictly sequential: vision → text → LLM. The LLM can't direct the vision system to attend to specific parts of the image, and the vision system can't be informed by the LLM's world knowledge about what's likely to matter in a given scene.

Vocabulary Mismatch

Language was designed to describe categories and concepts, not precise 3D geometry. When a robot manipulator needs sub-centimeter accuracy for grasping, verbal spatial descriptions ("a bit to the left") are wholly inadequate. The translation to language fundamentally loses geometric precision.

What PaLM-E Actually Proposes

Instead of this serial pipeline, PaLM-E's core proposal is simple but radical: treat images, robot states, and other sensory inputs as if they were language tokens. Map them into the same embedding space that the LLM already uses for words, then let the Transformer's self-attention mechanism do what it does best — reason over all available tokens jointly. The image "speaks" directly to the language reasoning engine, in the same representational currency.

§02 Prior Work Landscape: What Came Before

Understanding PaLM-E requires understanding the intellectual space it enters. Three prior families of work are most relevant: general vision-language models, action-output robot models, and LLM-based planners.

Vision-Language Models (VLMs)

Models like Flamingo (Alayrac et al., 2022) and CLIP showed that vision and language can be jointly trained at scale. Flamingo in particular augments a frozen LLM with cross-attention to visual features via "Perceiver Resampler" modules. The key limitation: these models handle perception well but were not designed for, and do not generalize to, embodied reasoning. The paper shows that state-of-the-art VLMs like PaLI achieve 0% accuracy on robotic manipulation planning tasks in a zero-shot setting.

⚠ Critical Result

PaLI, a state-of-the-art vision-language model, achieves 0.0% on zero-shot TAMP planning tasks. Strong visual understanding does not automatically translate to embodied reasoning. Something more is needed — specifically, training on embodied data and a representation that preserves geometric information.

SayCan — The Most Comparable Prior Approach

SayCan (Ahn et al., 2022) is the most direct precursor. It routes between an LLM's language score (does this action make semantic sense?) and a learned value function's affordance score (can this action be executed in the current state?). The LLM says what should be done next; a separate neural network says what is physically possible. The two scores are multiplied to select an action.

SayCan Strengths

Works with off-the-shelf LLMs without any retraining. The language model provides strong common-sense reasoning about task sequencing. Affordance functions prevent physically impossible steps.

SayCan Limitations

Requires oracle or separately-learned affordances for every skill. The LLM receives only text — it never actually "sees" the environment. Long-horizon task success rate remains low because affordance functions are not trained jointly with the planner.

Frozen LLM (Tsimpoukelli et al., 2021)

The "Frozen" approach trains only the visual encoder while keeping the LLM frozen. The encoder learns to produce a visual "soft prompt" that steers the LLM's text generation. PaLM-E is directly inspired by this idea but substantially generalizes it: multiple modalities, multiple encoder architectures (including 3D-aware ones), and empirically outperforms Frozen by over 45% on VQAv2 when the LLM is also fine-tuned.

Gato — The Generalist Agent

Gato (Reed et al., 2022) is the most architecturally similar prior work: a single model trained on many tasks including robotics, games, and language. The critical difference is transfer. Gato processes different tasks but does not demonstrate that training on one task helps another. PaLM-E explicitly shows and quantifies positive transfer — robot task performance significantly improves from internet-scale vision-language pretraining.

§03 The Formal Model: Math From First Principles

Let's derive PaLM-E's architecture from the mathematics of decoder-only LLMs, step by step.

Standard Decoder-Only LLM

A standard decoder-only LLM models a joint probability over a sequence of tokens. Given vocabulary W of size 256,000 and a sequence of tokens w₁, w₂, ..., wₗ, the model factors the joint as a product of conditionals:

Equation 1 — Autoregressive Factorization
p(w₁:L) = ∏ₗ₌₁ᴸ p_LM(wₗ | w₁:ₗ₋₁)

Each token wᵢ is mapped to a continuous vector via a learned embedding matrix γ: W → X ⊂ ℝᵏ. For PaLM-540B, k=18,432 (the hidden dimension). The embedding matrix has shape k × |W| = 18432 × 256000 ≈ 4.7 billion parameters just for the embedding layer alone.

Prefix-Decoder Conditioning

During inference (and the training of the multimodal extension), the model is conditioned on a prefix sequence w₁:ₙ. The probability of the generated completion is:

Equation 2 — Prefix Conditioning
p(wₙ₊₁:L | w₁:ₙ) = ∏ₗ₌ₙ₊₁ᴸ p_LM(wₗ | w₁:ₗ₋₁)

This formulation is unchanged — PaLM-E doesn't modify how the LLM generates text. What changes is what goes into the prefix.

The Multimodal Injection — Core Equation

The key innovation is Equation 3. Instead of all prefix vectors coming from the word embedding table γ, some positions are filled by a trained encoder φⱼ applied to a continuous observation Oⱼ:

Equation 3 — Multimodal Token Construction (Central to PaLM-E)
xᵢ = { γ(wᵢ) if position i is a text token { φⱼ(Oⱼ)ᵢ if position i corresponds to observation Oⱼ

The encoder φ: O → X^q maps a single observation (e.g., an image) into a sequence of q vectors, each of dimension k. These q vectors occupy q consecutive positions in the prefix. The LLM processes this extended prefix with the same self-attention mechanism it uses for pure text — no architectural changes.

🔑 Why This Is Elegant

The LLM's self-attention layers can attend to image patch vectors just as they attend to word vectors. "Chips" (a text token) can directly attend to the visual patch embedding of the chips bag in the image. Cross-modal reasoning happens implicitly through the Transformer's existing attention mechanism. The model learns to align visual features with linguistic concepts through gradient descent on the training objective — no manual feature engineering or hard-coded alignment rules.

Loss Function

During training, loss is computed only on the text output tokens (not on the multimodal prefix). Given a training example with observations O₁:ₘ, text prefix w₁:ₙ, and target text wₙ₊₁:L, the objective is standard cross-entropy:

Training Loss
L = − ∑ₗ₌ₙ₊₁ᴸ log p_LM(wₗ | x₁:ₗ₋₁)

where each xᵢ in the prefix is constructed using the mixed rule in Equation 3. Gradients flow back through the LLM (if not frozen) and through the encoders φⱼ, training them to produce embeddings that are useful for predicting the target tokens.

§04 Input Encoders: ViT, TokenLearner & State Vectors

PaLM-E experiments with three different visual encoder architectures, plus a simple MLP for state vectors. The choice of encoder has significant implications for the model's spatial understanding and computational efficiency.

ViT-4B and ViT-22B — Standard Patch Embeddings

The standard Vision Transformer (ViT) divides an image into a grid of non-overlapping patches (typically 14×14 patches for a 224×224 image with 16×16 patch size). Each patch is flattened and projected to a vector, then processed by Transformer layers with 2D position encodings. PaLM-E uses two scales:

ViT-4B SoViT
~4B params · Pre-trained on image classification · ~256 patch tokens
ViT-22B Largest Used
22B params · Used in PaLM-E-562B · Dehghani et al. 2023

The ViT output has dimensionality k̃ which is not necessarily the same as PaLM's hidden dimension k. A learned linear projection ψ: ℝᵏ̃ → ℝᵏ (affine transformation) maps each patch embedding to the language model's space:

ViT Projection
φ_ViT(I)ᵢ = ψ( φ̃_ViT(I)ᵢ )

A 224×224 image with 16×16 patches produces 196 patch tokens. This means injecting a single image adds 196 positions to the prefix sequence. For the 562B model, each position is an 18,432-dimensional vector.

ViT + TokenLearner — Learned Token Compression

196 tokens per image is expensive — each additional prefix token increases the self-attention computation quadratically. TokenLearner (Ryoo et al., 2021) addresses this by learning a small set of weighted combinations of patch features, compressing 196 spatial patches into as few as 8–32 learned tokens:

HOW TOKENLEARNER WORKS
Input: 196 ViT patch embeddings (196, k̃) TokenLearner learns S "selector" functions: αₛ(I) = softmax( MLP_s(patch_features) ) → attention weight per patch zₛ = ∑ᵢ αₛ(I)ᵢ · patch_embeddingᵢ → weighted combination Output: S learned token vectors (S, k̃) where S ≪ 196 Then project: S × k̃ → S × k (affine ψ) This is trained end-to-end with PaLM-E. The model learns which parts of the image matter.

The ViT+TL variant is trained from scratch alongside PaLM-E rather than initialized from a pre-trained ViT. This is notable: it suggests the model can learn useful visual representations jointly with the language task.

State Vector Encoding — The Simplest Modality

Robot state vectors (joint angles, end-effector pose, object positions from state estimation) are the simplest modality to inject. A state vector s ∈ ℝˢ is mapped to a small number of embedding vectors via an MLP:

class StateEncoder(nn.Module):
    """
    Maps a robot state vector s ∈ R^S to a sequence of q
    embedding vectors in the LLM's embedding space X ⊂ R^k.
    
    In PaLM-E, S is typically:
      - 7 DOF robot arm: 7 joint angles + 7 velocities + gripper = ~15 floats
      - Object states: (x, y, z, roll, pitch, yaw) × N_objects
      - In TAMP experiments: full factored state representation
    """
    def __init__(self, state_dim: int, embed_dim: int, num_tokens: int = 4):
        super().__init__()
        self.num_tokens = num_tokens
        self.mlp = nn.Sequential(
            nn.Linear(state_dim, embed_dim * 2),
            nn.SiLU(),                          # SiLU used in PaLM
            nn.Linear(embed_dim * 2, embed_dim * num_tokens),
        )

    def forward(self, state: torch.Tensor) -> torch.Tensor:
        # state: (B, S)
        B = state.shape[0]
        out = self.mlp(state)                             # (B, embed_dim * num_tokens)
        return out.view(B, self.num_tokens, -1)           # (B, num_tokens, embed_dim)

§05 OSRT: 3D Object-Centric Scene Representations

OSRT (Object Scene Representation Transformer, Sajjadi et al. 2022) is the most architecturally interesting encoder in PaLM-E, and the one that achieves the best robotic manipulation results. It deserves a detailed treatment.

The Problem With Grid-Based ViT Representations

ViT produces a grid of patch embeddings. This representation is semantically rich but spatially rigid — each embedding corresponds to a fixed 16×16 region of the image, regardless of where objects are. For a robot that needs to reason about objects — "pick up the red cube on the left and place it on top of the blue cylinder" — a grid representation has a mismatch: objects can span multiple patches, and the model must figure out which patches correspond to which object.

More critically, ViT is completely 2D. It has no understanding of depth, no 3D structure. For manipulation tasks where the robot needs to know if block A is actually behind block B or just appears to be in the image, this is a severe limitation.

How OSRT Works

OSRT Architecture: Multi-View → Objects → Slots ═══════════════════════════════════════════════════════════════ INPUT: v camera views of the same scene {I₁, I₂, ..., Iᵥ} Step 1: Encode each view with a shared CNN/ViT backbone features = encoder(Iⱼ) for j in 1..v Step 2: Object discovery via slot attention ┌─────────────────────────────────────────────────────────┐ │ Slot Attention (Locatello et al., 2020): │ │ Initialize N_obj random "slot" vectors │ │ Iteratively compete to explain different │ │ regions of the feature map via soft attention │ │ │ │ slots ← SlotAttention(features) │ │ Output: N_obj slot vectors, each capturing │ │ a distinct object or background element │ └─────────────────────────────────────────────────────────┘ Step 3: Novel view synthesis (training signal) Use NeRF-style decoder to render held-out views The scene representation must encode 3D geometry to correctly predict pixels from new viewpoints OUTPUT: N_obj object slot vectors ōⱼ ∈ R^k̄ Each slot ≈ one object in 3D space Project: ψ(ōⱼ) → ℝᵏ (LLM embedding dim) Inject these N_obj vectors into the prefix
🧠 Why OSRT Outperforms ViT on Robot Tasks

Three key advantages compound: (1) 3D awareness — by training to synthesize novel views, OSRT must infer 3D structure; (2) object-centricity — each token corresponds to a distinct object, making it natural for the LLM to reason "pick up object 3, place on object 1"; (3) compact representation — 5 objects → 5 tokens vs. 196 patch tokens for ViT, leading to a shorter prefix and less "noise" for the LLM to process. On TAMP tasks, OSRT achieves 98.2% VQA accuracy vs. 70.7% for ViT-4B.

Object Slot Handling in PaLM-E

When OSRT is used, the slot vectors are projected into the LLM's embedding space and injected as a compact set of vectors. Since OSRT uses a pre-trained slot representation, the PaLM-E training freezes the OSRT slot representations and only trains a small linear projection layer — this is a form of parameter-efficient adaptation. The full OSRT architecture is not fine-tuned.

Implementation detail from the paper: "In experiments with OSRT, the slot representation is also frozen, and only the small projector, which serves as the interface between OSRT and the LLM, is updated." This is an important distinction from the ViT encoders, which are jointly trained with PaLM.

§06 Entity-Labeling Multimodal Tokens: A Novel Contribution

Entity labeling is a novel architectural idea introduced in PaLM-E that significantly improves robotic manipulation planning. It addresses a subtle but critical problem: how does the LLM refer to a specific object in its output plan when that object can't be identified by simple language properties?

The Reference Problem

Imagine the scene has two green blocks of identical size. The LLM needs to plan "pick up block A and place it on block B." But "block A" and "block B" are not distinguishable by color, size, or any visual property captured in natural language. How does the model reference them?

The Entity-Labeling Solution

For object-centric representations (OSRT), each object slot is assigned a special token in both the input prompt and the expected output. The input format becomes:

Entity-Labeled Input Template: ──────────────────────────────────────────────────────────────── "Object 1 is <obj_1>. Object 2 is <obj_2>. Object 3 is <obj_3>. Task: Move the object that is on the left side to the top of the green cylinder. Plan:" Where <obj_1> is the OSRT slot vector for object 1, <obj_2> is the OSRT slot vector for object 2, etc. Expected output: "1. grasp(<obj_1>) 2. place_on(<obj_1>, <obj_3>)" The model must: (a) Identify which <obj_*> token corresponds to the "object on the left side" from the visual encoding (b) Output the correct special token in its plan (c) The low-level policy then maps <obj_*> tokens to actual robot trajectories
🔑 Why This Is Clever

The LLM's vocabulary is extended with special object reference tokens <obj_1>, <obj_2>, etc. These tokens appear in both the prefix (where they are replaced by OSRT slot vectors during the embedding step) and in the model's output (where they are predicted as vocabulary tokens). The training signal teaches the model to associate the visual slot representation with the corresponding reference token. This creates a bidirectional grounding: visual features → reference token (perceptual) AND reference token → visual features (generative).

Training with Entity Labels

The training data for TAMP tasks must be constructed with entity labels. Each demonstration trajectory specifies which object slot each step acts on. The data pipeline constructs prompts in the entity-labeled format, and the model is trained with cross-entropy to predict the correct sequence of labeled actions.

This is a significant data engineering challenge: the entity labels must be consistent (same object always gets the same slot index across a trajectory) and the OSRT model must have consistent slot assignments across frames. Object permanence in the slot assignment is non-trivial and requires careful handling in the OSRT pre-processing pipeline.

§07 Training: Data Mixture, Scale & Methodology

PaLM-E's training is a multi-task mixture across radically different domains. Understanding the data mix is essential to understanding why positive transfer occurs.

The Full Training Mixture

The full training mixture is dominated by internet-scale vision-language data, with embodied robotics data making up only a small fraction. The key numbers from the paper:

📊 Critical Statistic

In the full training mixture, only 8.9% of examples are embodied robotics data. The remaining ~91% are general vision-language tasks (VQA, captioning, etc.) and pure language tasks. Yet this mixture leads to robot task performance that far exceeds training on robot data alone. This is the empirical heart of positive transfer.

Vision-Language Tasks (~82%)

Visual Question Answering (VQAv2), OK-VQA (knowledge-intensive VQA), image captioning, visual dialogue, scene understanding tasks. These provide the core grounding of visual features to language.

Pure Language Tasks (~9%)

Standard language modeling on web text, books, and other corpora from PaLM's original pretraining distribution. This preserves the LLM's core language capabilities and prevents catastrophic forgetting of language.

TAMP Robotics (subset of 8.9%)

Task and Motion Planning in simulation. 320 examples for each of two planning tasks (p1, p2). 640 total manipulation planning examples. This is an extremely data-sparse regime — fewer than a thousand training demonstrations.

Language-Table & SayCan Mobile (~8.9% total)

Language-Table tabletop pushing (10–80 examples for few-shot experiments). SayCan mobile manipulation. Real-world closed-loop environments. Training examples here consist of (image, instruction, next_skill) triples.

Training Regimen

PaLM-E is initialized from pre-trained weights for both the LLM and vision encoder components. The training procedure has several important design choices:

1

Initialization — Start Strong

PaLM-E starts from a pre-trained PaLM (8B, 62B, or 540B depending on model size) and a pre-trained ViT (4B or 22B pre-trained on image classification). Both bring substantial knowledge before any joint training begins. The linear projection layer ψ mapping ViT → LLM embedding space is initialized randomly.

2

Joint Multitask Training — One Objective

All tasks are trained with the same cross-entropy loss on the text output tokens. The model sees a random sample from the full mixture at each step. No task-specific heads, no curriculum scheduling in the base variant. The loss mask excludes the multimodal prefix tokens and only trains on the target text.

3

Sampling Frequencies

Task sampling frequencies are manually tuned, not learned. Embodied tasks are up-sampled relative to their natural frequency to ensure the model doesn't ignore the small robotics dataset. The exact sampling ratios are reported in Appendix D of the paper. Getting these weights right is critical for the quality of positive transfer.

4

Hardware: TPU v4 at Scale

The 562B model training was conducted on Google's TPU v4 infrastructure, inheriting the parallelism strategy from PaLM's original training (2× TPU v4 Pods, 6,144 chips total, model + data parallelism). Training the full 562B model with the multimodal encoders is a significant engineering effort not detailed fully in the paper.

§08 Freeze vs. Fine-Tune: A Critical Design Choice

One of the paper's key ablations studies what to freeze during training. This has direct implications for how you'd approach training your own multimodal LLM.

The Three Architectural Components

PaLM-E has three trainable components: the encoder φ̃ (ViT), the projector ψ (linear layer), and the LLM p_LM (PaLM). Any subset can be frozen.

Strategy A: Frozen LLM + Train Encoder + Projector

Only the visual encoder and projection layer are updated. The LLM remains at its pretrained weights. This is "input-conditioned soft prompting" — the encoder must produce prompts that steer the frozen LLM correctly. Advantage: guaranteed no catastrophic forgetting of language tasks. Disadvantage: encoder must overcome all the mismatch between visual features and language representations alone.

Strategy B: Full Fine-Tuning (Winner)

All components are updated. The LLM can adapt its internal representations to be more conducive to multimodal reasoning. The encoder can be guided by gradients from both the LLM's predictions and its own feature learning. This achieves significantly better performance on robotics tasks but risks catastrophic forgetting.

⚠ Trade-off Summary

The paper shows that full fine-tuning consistently outperforms frozen LLM on all robotic tasks. However, frozen LLM incurs zero catastrophic forgetting on language benchmarks. The choice depends on your priorities: if preserving language capabilities is critical, freeze the LLM and accept some performance loss on embodied tasks. If you're building a dedicated robotics system, full fine-tuning is better.

Soft Prompting vs. Full Fine-Tuning — Technical Analysis

When the LLM is frozen, the visual encoder must produce representations that are already compatible with the LLM's frozen attention patterns. This is a significantly harder task: the encoder must produce prompt vectors that lie in the distribution of "text that naturally precedes the desired output." In practice, this means the encoder needs to be larger and more expressive to compensate.

When the LLM is jointly fine-tuned, gradients flow back through the LLM's layers to shape the encoder's representations. The LLM can simultaneously learn to "read" visual embeddings and the encoder can learn to produce embeddings that the (now adapted) LLM finds useful. This mutual adaptation is why full fine-tuning wins.

OSRT-Specific Freezing

For OSRT experiments, the slot representation itself is frozen and only the projector is trained. This is a practical decision: training OSRT jointly with a 562B LLM would be computationally prohibitive, and the OSRT slots are already informative 3D-aware representations that only need to be adapted (projected) into the LLM's space.

§09 PaLM-E in the Robot Control Loop

PaLM-E operates as a high-level policy in a hierarchical control architecture. It does not directly produce motor commands; instead, it produces natural language instructions that are executed by low-level learned policies.

The Control Architecture

PaLM-E Robot Control Architecture ═════════════════════════════════════════════════════════════════ Human Task: "Bring me the rice chips" │ ▼ ┌─────────────────────────────────────────────────────────────┐ │ PaLM-E (562B) │ │ INPUT: Text instruction + RGB camera images │ │ + robot state vector (joint angles, etc.) │ │ │ │ Processes: Multimodal sentence through 118 PaLM layers │ │ Outputs: "Step 1: navigate_to(drawer_area)" │ │ "Step 2: open_drawer(top)" │ │ "Step 3: grasp(rice_chip_bag)" │ └──────────────────────────┬──────────────────────────────────┘ │ text plan ▼ ┌──────────────────────────────────────────────────────────────┐ │ Low-Level Policy (e.g., RT-1) │ │ Receives: "navigate_to(drawer_area)" │ │ Produces: 768-dim action token per timestep (RT-1) │ │ → decoded to: delta_xyz, delta_rpy, gripper │ │ Executes for ~20-50 timesteps until skill completes │ └──────────────────────────┬───────────────────────────────────┘ │ new observation ▼ ┌──────────────────────────────────────────────────────────────┐ │ Environment: Robot + Physical World │ │ New RGB images + updated joint states returned │ │ (closed-loop: PaLM-E replans if execution fails) │ └──────────────────────────┬───────────────────────────────────┘ │ └──→ back to PaLM-E with new obs

Three Robot Environments

TAMP (Simulation)

Task and Motion Planning environment. Robot must grasp and stack objects. 3-5 objects per scene. Ground-truth state estimation available. Uses OSRT encoding. Tests multi-step plans requiring geometric understanding.

Language-Table (Real)

Tabletop pushing environment. 8 objects. Closed-loop with real robot. Language-conditioned goals. Uses RGB images. Few-shot learning regime: trained on only 10-80 examples per skill, must generalize to novel objects.

SayCan Mobile (Real)

Mobile manipulator in kitchen environment. Must navigate to objects, open containers, grasp items. Long-horizon tasks (5-10+ steps). Affordance and failure detection evaluated. Closest to real-world deployment.

Replanning Under Failure

PaLM-E's closed-loop integration enables replanning. After each skill execution, new camera images are captured and fed back into PaLM-E along with the history of completed steps. If a skill failed (e.g., the gripper missed the object), PaLM-E can observe the failure and adjust its plan. This is evaluated in the mobile manipulation experiments where PaLM-E must also perform failure detection — identifying when a step didn't succeed and deciding whether to retry or replan.

🔄 Closed-Loop vs. Open-Loop Planning

The full multimodal sentence at each replanning step contains the original task, the full history of completed steps and their outcomes, and the new observation. The context window must be long enough to hold this growing history. This is a form of in-context learning — PaLM-E adapts its plan based on new evidence without any weight updates at test time.

def precompute_freqs_cis(dim: int, seq_len: int, theta: float = 10000.0):
    freqs = 1.0 / (theta ** (torch.arange(0, dim, 2).float() / dim))
    t     = torch.arange(seq_len)
    freqs = torch.outer(t, freqs)
    return torch.polar(torch.ones_like(freqs), freqs)  # complex64

def apply_rope(x: torch.Tensor, freqs_cis: torch.Tensor) -> torch.Tensor:
    x_c = torch.view_as_complex(x.float().reshape(*x.shape[:-1], -1, 2))
    return torch.view_as_real(x_c * freqs_cis).flatten(-2).type_as(x)


# ─────────────────────────────────────────────────────────────────────────
# 2.  MULTI-QUERY ATTENTION — PaLM uses a single KV head for efficiency
#     Multiple query heads, one shared K and V head.
#     Reduces KV cache memory at inference by factor of n_heads.
# ─────────────────────────────────────────────────────────────────────────
class MultiQueryAttention(nn.Module):
    def __init__(self, dim: int, n_heads: int):
        super().__init__()
        self.n_heads  = n_heads
        self.head_dim = dim // n_heads
        # Q: one projection per head; K, V: one shared projection
        self.q_proj = nn.Linear(dim, dim, bias=False)
        self.k_proj = nn.Linear(dim, self.head_dim, bias=False)
        self.v_proj = nn.Linear(dim, self.head_dim, bias=False)
        self.o_proj = nn.Linear(dim, dim, bias=False)

    def forward(self, x, freqs_cis, mask=None):
        B, T, C = x.shape
        Q = self.q_proj(x).view(B, T, self.n_heads, self.head_dim).transpose(1,2)
        K = self.k_proj(x).view(B, T, 1, self.head_dim).transpose(1,2)
        V = self.v_proj(x).view(B, T, 1, self.head_dim).transpose(1,2)
        # Apply RoPE to Q and K
        Q = apply_rope(Q, freqs_cis)
        K = apply_rope(K, freqs_cis)
        # Expand K, V to all heads (broadcast)
        K = K.expand(B, self.n_heads, T, self.head_dim)
        V = V.expand(B, self.n_heads, T, self.head_dim)
        scale = self.head_dim ** -0.5
        attn  = (Q @ K.transpose(-2,-1)) * scale
        if mask is not None:
            attn = attn.masked_fill(mask, float('-inf'))
        attn = F.softmax(attn, dim=-1)
        out  = (attn @ V).transpose(1,2).reshape(B, T, C)
        return self.o_proj(out)


# ─────────────────────────────────────────────────────────────────────────
# 3.  SWIGLU FEED-FORWARD — PaLM's FFN activation
#     SwiGLU(x) = SiLU(W₁x) ⊙ W₃x   (gated; W₂ projects down)
#     Uses 2 × up-projections rather than 1 — empirically better.
# ─────────────────────────────────────────────────────────────────────────
class SwiGLUFFN(nn.Module):
    def __init__(self, dim: int, ffn_dim: int):
        super().__init__()
        self.w1 = nn.Linear(dim, ffn_dim, bias=False)  # gate
        self.w3 = nn.Linear(dim, ffn_dim, bias=False)  # value
        self.w2 = nn.Linear(ffn_dim, dim, bias=False)  # down proj

    def forward(self, x):
        return self.w2(F.silu(self.w1(x)) * self.w3(x))


# ─────────────────────────────────────────────────────────────────────────
# 4.  TRANSFORMER BLOCK — pre-norm with RMS Norm (not LayerNorm)
# ─────────────────────────────────────────────────────────────────────────
class TransformerBlock(nn.Module):
    def __init__(self, dim: int, n_heads: int, ffn_dim: int):
        super().__init__()
        self.attn_norm = nn.RMSNorm(dim)
        self.attn      = MultiQueryAttention(dim, n_heads)
        self.ffn_norm  = nn.RMSNorm(dim)
        self.ffn       = SwiGLUFFN(dim, ffn_dim)

    def forward(self, x, freqs_cis, mask=None):
        x = x + self.attn(self.attn_norm(x), freqs_cis, mask)
        x = x + self.ffn(self.ffn_norm(x))
        return x


# ─────────────────────────────────────────────────────────────────────────
# 5.  IMAGE ENCODER — mimics ViT + linear projector ψ
#     Real ViT: patch_embed → positional encoding → N transformer layers
#     → 196 output patch vectors → project to EMBED_DIM
#     Here: simplified patch extraction + toy transformer
# ─────────────────────────────────────────────────────────────────────────
class ImageEncoder(nn.Module):
    def __init__(self, vit_dim: int, embed_dim: int, n_tokens: int):
        super().__init__()
        patch_dim = 3 * 8 * 8  # toy 8×8 RGB patches
        self.patch_embed  = nn.Linear(patch_dim, vit_dim)
        self.vit_layers   = nn.ModuleList([
            nn.TransformerEncoderLayer(vit_dim, nhead=4, batch_first=True, dim_feedforward=vit_dim*4)
        ])
        # TokenLearner: compress IMG_PATCHES → n_tokens learned vectors
        self.token_learner = nn.Sequential(
            nn.Linear(vit_dim, n_tokens),
            nn.Sigmoid()             # attention weights
        )
        # Projector ψ: maps ViT dim → LLM embedding dim
        self.projector = nn.Linear(vit_dim, embed_dim)
        self.n_tokens  = n_tokens

    def forward(self, img: torch.Tensor) -> torch.Tensor:
        # img: (B, 3, 32, 32)
        B = img.shape[0]
        patches = img.unfold(2,8,8).unfold(3,8,8)           # (B,3,4,4,8,8)
        patches = patches.permute(0,2,3,1,4,5).reshape(B, 16, -1) # (B,16,192)
        h = self.patch_embed(patches)                          # (B,16,vit_dim)
        for layer in self.vit_layers:
            h = layer(h)
        # TokenLearner: compute attention weights over patches, pool
        attn_w = self.token_learner(h).transpose(1,2)         # (B,n_tokens,16)
        attn_w = F.softmax(attn_w, dim=-1)
        tokens = attn_w @ h                                    # (B,n_tokens,vit_dim)
        return self.projector(tokens)                          # (B,n_tokens,embed_dim)


# ─────────────────────────────────────────────────────────────────────────
# 6.  STATE ENCODER — maps robot/scene state vector → embedding tokens
# ─────────────────────────────────────────────────────────────────────────
class StateEncoder(nn.Module):
    def __init__(self, state_dim: int, embed_dim: int, n_tokens: int):
        super().__init__()
        self.n_tokens = n_tokens
        self.mlp = nn.Sequential(
            nn.Linear(state_dim, embed_dim * 2),
            nn.SiLU(),
            nn.Linear(embed_dim * 2, embed_dim * n_tokens),
        )

    def forward(self, state: torch.Tensor) -> torch.Tensor:
        B = state.shape[0]
        return self.mlp(state).view(B, self.n_tokens, -1)      # (B,n_tokens,embed_dim)


# ─────────────────────────────────────────────────────────────────────────
# 7.  PaLM-E FULL MODEL
#     Assembles: multimodal sentence → LLM → text logits
# ─────────────────────────────────────────────────────────────────────────
class PaLME(nn.Module):
    def __init__(self):
        super().__init__()
        # ── Language components ────────────────────────────────────────────
        self.word_embed    = nn.Embedding(VOCAB_SIZE, EMBED_DIM)
        self.transformer   = nn.ModuleList([
            TransformerBlock(EMBED_DIM, N_HEADS, FFN_DIM) for _ in range(N_LAYERS)
        ])
        self.norm_out      = nn.RMSNorm(EMBED_DIM)
        self.lm_head       = nn.Linear(EMBED_DIM, VOCAB_SIZE, bias=False)
        # Tie embedding weights (standard practice in LLMs)
        self.lm_head.weight = self.word_embed.weight

        # ── Multimodal encoders ─────────────────────────────────────────────
        self.img_encoder   = ImageEncoder(VIT_DIM, EMBED_DIM, IMG_TOKS)
        self.state_encoder = StateEncoder(STATE_DIM, EMBED_DIM, STATE_TOKS)

        # ── Precompute RoPE frequencies ─────────────────────────────────────
        self.register_buffer(
            'freqs_cis',
            precompute_freqs_cis(EMBED_DIM // N_HEADS, 512)  # max_seq 512
        )

    def build_multimodal_prefix(self,
                                  text_ids: torch.Tensor,
                                  image: Optional[torch.Tensor] = None,
                                  state: Optional[torch.Tensor] = None
                                  ) -> Tuple[torch.Tensor, int]:
        """
        Build the multimodal prefix by interleaving text embeddings with
        encoded continuous observations, per Equation 3 in the paper.
        
        Template: [TEXT_BEFORE_IMG][IMAGE_TOKENS][STATE_TOKENS][TEXT_AFTER]
        
        Returns: (prefix_embeddings, prefix_length)
        prefix_embeddings: (B, T_total, EMBED_DIM)
        """
        parts = []

        # Text prefix (e.g., "Human: Bring me rice chips. Robot: I see")
        text_emb = self.word_embed(text_ids)                  # (B, T_text, D)
        parts.append(text_emb)

        # Image tokens (e.g., "I see [IMAGE_TOKEN_1...IMAGE_TOKEN_8]")
        if image is not None:
            img_emb = self.img_encoder(image)                 # (B, IMG_TOKS, D)
            parts.append(img_emb)

        # State tokens (e.g., "State: [STATE_TOKEN_1...STATE_TOKEN_4]")
        if state is not None:
            state_emb = self.state_encoder(state)             # (B, STATE_TOKS, D)
            parts.append(state_emb)

        prefix = torch.cat(parts, dim=1)                     # (B, T_prefix, D)
        return prefix, prefix.shape[1]

    def forward(self,
                  text_ids: torch.Tensor,
                  target_ids: torch.Tensor,
                  image: Optional[torch.Tensor] = None,
                  state: Optional[torch.Tensor] = None
                  ) -> torch.Tensor:
        """
        Forward pass for training.
        Loss is computed ONLY on target_ids tokens (not the multimodal prefix).
        
        text_ids:   (B, T_prefix_text)   — prefix text token IDs
        target_ids: (B, T_target)        — gold output token IDs
        image:      (B, 3, H, W)
        state:      (B, state_dim)
        """
        B = text_ids.shape[0]

        # Build multimodal prefix
        prefix_emb, T_prefix = self.build_multimodal_prefix(text_ids, image, state)

        # Embed target tokens and concatenate (teacher forcing)
        target_emb  = self.word_embed(target_ids[:, :-1])     # shift right
        full_input  = torch.cat([prefix_emb, target_emb], dim=1)
        T_total     = full_input.shape[1]

        # Causal attention mask
        mask = torch.triu(torch.ones(T_total, T_total, dtype=torch.bool), diagonal=1)

        # Run transformer
        freqs = self.freqs_cis[:T_total]
        x = full_input
        for block in self.transformer:
            x = block(x, freqs, mask)
        x = self.norm_out(x)

        # Logits for TARGET positions only (exclude the prefix)
        logits = self.lm_head(x[:, T_prefix:])                 # (B, T_target-1, VOCAB)
        # Cross-entropy loss — key: only penalise text output tokens
        loss = F.cross_entropy(
            logits.reshape(-1, VOCAB_SIZE),
            target_ids[:, 1:].reshape(-1),                   # shift targets
            ignore_index=0                                     # mask padding
        )
        return loss

    @torch.no_grad()
    def generate(self,
                    text_ids: torch.Tensor,
                    image: Optional[torch.Tensor] = None,
                    state: Optional[torch.Tensor] = None,
                    max_new_tokens: int = 20,
                    temperature: float = 0.7
                    ) -> List[int]:
        """
        Autoregressive generation given a multimodal prompt.
        Returns a list of generated token IDs.
        """
        prefix_emb, T_prefix = self.build_multimodal_prefix(text_ids, image, state)
        context_emb = prefix_emb                               # start with prefix
        generated   = []

        for _ in range(max_new_tokens):
            T = context_emb.shape[1]
            mask = torch.triu(torch.ones(T, T, dtype=torch.bool), diagonal=1)
            freqs = self.freqs_cis[:T]
            x = context_emb
            for block in self.transformer:
                x = block(x, freqs, mask)
            x = self.norm_out(x)
            # Sample from last position only
            logits   = self.lm_head(x[:, -1]) / temperature
            probs    = F.softmax(logits, dim=-1)
            next_tok = torch.multinomial(probs, 1)
            generated.append(next_tok.item())
            # Append new token embedding to context
            next_emb = self.word_embed(next_tok)
            context_emb = torch.cat([context_emb, next_emb], dim=1)

        return generated

The most practically relevant benchmark: a mobile robot must complete multi-step household tasks. PaLM-E evaluates three capabilities in this environment:

Affordance Prediction

Given an image and a proposed skill, is the skill currently executable? PaLM-E learns to predict this from visual context, enabling physically grounded planning without a separately trained affordance function.

Did the last skill succeed? PaLM-E can observe the current state and the expected post-condition and detect failures, enabling replanning. This is a key capability SayCan doesn't have.

Long-Horizon Planning

5-10+ step task sequences in a real kitchen. PaLM-E conditions on the current image at each step, enabling closed-loop recovery from failures. Full fine-tune significantly outperforms frozen LLM here.

§12 Positive Transfer: The Paper's Central Empirical Claim

Positive transfer — where training on Task A improves performance on Task B — is the headline empirical contribution of PaLM-E. It's worth examining precisely what transfer occurs and why the authors believe it happens.

Transfer Type 1: VL Data → Robot Tasks

Training on internet-scale vision-language data (VQA, captioning) significantly improves robotic manipulation performance, even though there is essentially zero topical overlap — VQA images are not robot scenes, and captioning tasks don't involve planning. Why does this transfer work?

🔬 Mechanistic Hypothesis

Internet-scale VL training forces the visual encoder and projection layer to learn general-purpose visual feature extraction — understanding of objects, spatial relationships, colors, textures, and fine-grained visual details. This learned visual vocabulary is precisely what robotic manipulation requires. When the robot task says "the green block is to the left of the red cylinder," the visual understanding was already developed from VQA examples like "What color is the object on the left?". The transfer is not topical but representational: the same visual features that help answer VQA questions about everyday images also help localize objects in robot environments.

Transfer Type 2: TAMP VQA → TAMP Planning

Within the robot domain, including TAMP VQA (answering spatial questions about the scene) alongside TAMP planning (generating action sequences) improves planning performance. This is a tighter form of transfer: understanding what is where in the scene (a perceptual question) helps generate physically grounded plans (a sequential decision task). The paper ablates this by removing TAMP VQA from the training mix and showing a performance drop on planning.

Transfer Type 3: Language → All Tasks

Including pure language data in the training mixture prevents the model from forgetting the language capabilities of the base LLM. But there's also a positive signal: the reasoning patterns, common-sense knowledge, and linguistic structure learned from text data generalize to the generation of well-formed plans and coherent answers. The model doesn't just "remember" language — it applies language-learned reasoning strategies to multimodal tasks.

§13 Catastrophic Forgetting: Scale as the Solution

Catastrophic forgetting — where learning new tasks overwrites old knowledge — is one of the most fundamental challenges in fine-tuning large models. PaLM-E provides one of the most striking demonstrations of how model scale directly mitigates this problem.

Measuring Language Ability Retention

The paper evaluates language ability using two standard benchmarks: HellaSwag (common sense reasoning) and WinoGrande (commonsense disambiguation). These are pure language tasks evaluated after multimodal fine-tuning. The metric is retention relative to the base PaLM model.

Why Does Scale Help?

Several hypotheses are plausible, and the paper doesn't definitively resolve this question. But the most compelling explanation involves the ratio of new information to stored information:

Hypothesis 1: Capacity Surplus

Larger models have vastly more parameters per unit of training data. A 562B model trained on the same number of examples as a 12B model has more "room" to store new knowledge without overwriting old representations. The new visual grounding pathways don't need to repurpose neurons from the language capabilities.

Hypothesis 2: Optimization Landscape

The loss landscape of larger models tends to be flatter and better-conditioned. Small perturbations from fine-tuning (the gradient steps that learn multimodal grounding) cause less displacement in the original loss landscape, so language capabilities are less disrupted.

Hypothesis 3: Redundant Representations

Larger models develop more redundant, distributed representations. Language knowledge is stored across many overlapping circuits. When some circuits are partially repurposed for multimodal reasoning, the remaining circuits still adequately encode the language knowledge. Smaller models have no such redundancy.

Practical Implication

If you want to add a new modality to an LLM without degrading its original capabilities, use a larger base model. This is counterintuitive from a compute-efficiency standpoint, but the forgetting penalty on small models can make them practically unusable as general assistants after multimodal fine-tuning.

Role of the Training Mixture

Including pure language examples in the training mixture (the ~9% language-only data) significantly helps retention even for smaller models. This is a form of experience replay: regularly re-exposing the model to language-only examples prevents the optimization from drifting too far from language-compatible representations. The trade-off is training time — more data types to cover — but the authors find it worthwhile even for the 562B model.

§14 Legacy, Context & What Came After

What PaLM-E Got Right (That Others Later Adopted)

PaLM-E's core architecture — interleaving visual tokens into the LLM's embedding space — became the dominant paradigm for multimodal language models. Models like LLaVA (Liu et al., 2023), InstructBLIP, CogVLM, GPT-4V, and Google Gemini all use variations of this approach. The specific contributions that propagated:

✦

Visual Token Injection as the Standard Paradigm

The idea that visual encoders should project to the LLM's embedding dimension and their outputs should be placed directly in the token sequence — rather than through cross-attention or adapter mechanisms — became the default design choice. LLaVA uses exactly this architecture with MLP projection instead of linear.

✦

Positive Transfer as Design Principle

PaLM-E's empirical proof that diverse non-robotics data helps robot tasks was foundational for RT-2 (Brohan et al., 2023), which trained a VLA directly on internet-scale VL data and dramatically improved robot generalization. The principle — scale and diversity of training data matters more than direct task relevance — is now widely accepted in embodied AI.

✦

Scale as an Anti-Forgetting Strategy

The finding that 562B parameters retain language capability while 12B loses 87% has directly influenced how practitioners think about multimodal fine-tuning. It motivated LoRA and adapter-based approaches as memory-efficient alternatives for smaller models that can't rely on scale for retention.

✦

Zero-Shot Multimodal Chain-of-Thought

PaLM-E-562B's ability to perform zero-shot multimodal CoT despite being trained only on single-image examples prefigured the broader finding that sufficiently large and capable VLMs exhibit emergent reasoning behaviors. This contributed to the research agenda around eliciting and understanding emergent multimodal capabilities.

Limitations and What Came Next

Compute Scale is Prohibitive

562B parameters with a 22B ViT requires hundreds to thousands of A100/H100 GPUs just for inference. RT-2 and subsequent work pursued more practical scales (7B-70B). PaLM-E demonstrated the ceiling; production systems operate far lower.

Text-Only Output Limits Action Expressiveness

Generating skill names as text requires a vocabulary of enumerated low-level skills. RT-2 addresses this by predicting robot actions directly as tokenized continuous values in the output stream. End-to-end action generation avoids the bottleneck of a pre-defined skill library.

No Open Weights

PaLM-E was never released as an open-source model, limiting reproducibility and downstream research. The lack of open weights was a significant limitation compared to open VLM ecosystems that developed later (LLaVA, Idefics, etc.).

Single-Image Training Limitation

Despite emergent multi-image reasoning, the model was trained on single-image examples. Explicit multi-image training (as in Gemini 1.5 and GPT-4V fine-tuning) achieves substantially better temporal and comparative visual reasoning for robotics tasks requiring video or multi-view inputs.

Contextual Significance in 2023

PaLM-E appeared at an inflection point. GPT-4 launched the same month (March 2023), and the broader AI community was grappling with what scaled LLMs could and couldn't do. PaLM-E's demonstration that a single large model could simultaneously handle VQA, robot planning, and language tasks — while achieving SOTA on OK-VQA — was a compelling proof-of-concept for the generalist model paradigm. It helped shift the debate in robotics from "which specialized architecture should we use for perception/planning/control?" to "how do we train large generalist models that subsume all these functions?"

"The result is a robot that can understand nuanced human instructions, reason about complex multi-step plans, and even crack jokes about what it sees in a camera" — a capability profile that would have been inconceivable from any single architecture just two years earlier.

Key Takeaways for Practitioners

1. Inject, Don't Describe

Skip the caption-then-reason pipeline. Map visual features directly to embedding space. Let the model learn its own visual language.

2. Embrace Diverse Data

For data-scarce domains (robotics, medical imaging), internet-scale task-adjacent data drives most of the performance. Don't over-optimize on domain data alone.

3. 3D Beats 2D for Manipulation

OSRT's 3D object-centric representation outperforms ViT grid representations on robot manipulation. Geometry matters — choose encoders that preserve it.

4. Scale Protects Language

If preserving general capabilities matters, go bigger. Small models multimodally fine-tuned become narrow specialists. Large models remain generalists.

5. Hierarchical is Practical

High-level language planning + low-level reactive policies is more sample-efficient and interpretable than end-to-end action prediction at PaLM scale.

6. Entity Labels for Reference

When objects can't be distinguished linguistically, special reference tokens in the vocabulary allow the model to produce grounded output that references specific visual instances.
