Component 1 — The Two Tokenizers
💡 Key design tension: Pure semantic tokenizers (like HuBERT) lack acoustic expressiveness. Pure acoustic tokenizers (like EnCodec) inject too much low-level detail, making LM modeling hard and causing long-horizon error accumulation. Qwen3-TTS solves this with two purpose-built tokenizers, each targeting a different latency/quality trade-off.
Qwen-TTS-Tokenizer-25Hz Single Codebook Semantic+Acoustic
Built on top of Qwen2-Audio — a pretrained audio understanding model. Two training stages inject both semantic and acoustic info into a single VQ codebook.

Component 2 — Dual-Track LM Architecture
The Core Language Model — Qwen3 backbone (0.6B or 1.7B)
❓ What is "dual-track"? Text tokens and speech tokens are processed together in a single autoregressive sequence. The LM sees both the text it must speak and the speech tokens already generated — allowing it to immediately decode text tokens into acoustic tokens frame by frame. No separate acoustic model step.

SPEAKER ENCODER (jointly trained)
A learnable speaker encoder is trained end-to-end with the LM. Given 3 seconds of reference audio, it produces a speaker embedding prepended to the input sequence. This enables the LM to condition its speech token predictions on the enrolled speaker's identity without any fine-tuning.
✅ Voice Cloning: Extract speaker embedding from 3-sec enrolled audio → prepend to sequence → LM clones that voice.
🎨 Voice Design: Describe a voice in natural language (e.g., "deep, warm, slightly husky male voice") — no reference audio needed. The Qwen3 text model understands the description and guides synthesis.

Component 3 — 6-Stage Training Pipeline
Pre-training (3 stages) + Post-training (3 stages)
PRE-TRAINING
S1
General Stage
5M+ hours multilingual speech across 10 languages. Builds monotonic text→speech token mapping. Establishes baseline TTS capability.
→
S2
High-Quality Stage
Continual pretraining on filtered high-fidelity data. Reduces hallucinations and artifacts. Improves output quality significantly.
→
S3
Long-Context Stage
Max token length: 8,192 → 32,768. Upsample long speech data. Enables 10+ minute synthesis without degradation.
POST-TRAINING (Alignment)
P1
DPO
Direct Preference Optimization. Construct preference pairs from human feedback on multilingual speech samples (winner vs. loser). Aligns with human quality perception.
→
P2
GSPO + Rule-Based Rewards
Group Sequence Policy Optimization. Rule-based rewards for verifiable properties (e.g. content accuracy, prosody stability). Improves robustness across tasks and languages.
→
P3
Speaker Fine-Tuning
Lightweight adaptation on specific target speakers. Produces CustomVoice variants. Further improves naturalness, expressiveness, and controllability for deployed voices.
💡 Thinking pattern: The VoiceDesign variant uses a probabilistically activated "thinking" mode during training — occasionally reasoning about how to interpret the voice description before generating speech tokens. This makes instruction-following more robust for complex or ambiguous voice descriptions.
Component 4 — Speech Control Modes
Voice Cloning 3-second enrolled audio
Two mechanisms, depending on whether you have text for the enrolled audio:
Method A: Speaker Embedding
Encode 3-sec audio → speaker embedding → prepend to input. Works without a transcript. Real-time streaming compatible.
Method B: In-context Learning
Provide text-speech pair from target speaker. The LM uses it as a few-shot example in its context window to infer prosody, style, and pace — identical to VALL-E's approach.
Voice Design Natural language description
Create entirely new voices from text descriptions — no reference audio required.
Example instruction prompt:
"A mature female presenter with a clear, warm tone, speaking at a moderate pace with slight British accent and confident energy."
Qwen3's deep text understanding interprets the description and guides the speaker embedding generation accordingly. The optional "thinking" mode helps with complex or ambiguous descriptions.
Fine-Grained Control ChatML instructions
Instructions are prepended to the input as a system/user prompt in ChatML format:
Controllable attributes:
• Speaking rate (slow / moderate / fast) • Energy / emphasis level • Emotion (happy, sad, angry, calm…) • Pauses and phrasing • Timbre / voice character
📝 The Qwen3 LM backbone already understands natural language — no special training for each attribute. Just describe what you want.
Implementation — Pseudocode
12Hz Inference Path (LM backbone)
# Qwen3-TTS-12Hz inference
def
synthesize_12hz(text, enrolled_audio_3sec, instruction=None):
# Step 1: Encode speaker identity
speaker_emb = speaker_encoder(enrolled_audio_3sec)

# Step 2: Build input sequence
tokens = [
chatml_prompt(instruction), # optional control
speaker_emb, # speaker identity
qwen_tokenizer(text), # BPE text tokens
]

# Step 3: Autoregressive generation (cb0 only)
cb0_tokens = []
while True:
cb0_next = lm_backbone(tokens + cb0_tokens)
if cb0_next == EOS: break
cb0_tokens.append(cb0_next)

# Step 4: MTP fills remaining 15 codebooks (parallel)
cb1_to_15 = mtp_module(cb0_tokens) # [T, 15]
all_codes = stack([cb0_tokens, cb1_to_15], dim=1) # [T, 16]

# Step 5: Decode to waveform (causal ConvNet)
waveform = causal_convnet_decoder(all_codes)
return waveform
25Hz Inference Path (streaming)
# Qwen3-TTS-25Hz streaming inference
def
synthesize_25hz_stream(text, enrolled_audio, instruction=None):
# Same input sequence construction...
speech_tokens = []
chunk_buffer = []

while True:
# Single codebook — one token at a time
token = lm_backbone_linear_head(input)
if token == EOS: break
speech_tokens.append(token)
chunk_buffer.append(token)

# Emit audio after 8 tokens (need 16 for FIRST chunk)
if len(chunk_buffer) == 8:
mel = dit_decoder(chunk_buffer, context) # block-wise DiT
wav = bigvgan(mel) # vocoder
yield wav # stream audio packet
chunk_buffer = []
⚠️ Important: The 25Hz DiT needs a 1-block lookahead. This means you must buffer 16 LM tokens before emitting the first audio chunk (twice the chunk size of 8), adding ~190ms to first-packet latency.

Key Architectural Differences from VALL-E
How Qwen3-TTS advances on VALL-E's design
AspectVALL-EQwen3-TTS
Tokenizer EnCodec (RVQ, 8 codebooks) Two purpose-built tokenizers: 25Hz single-VQ (semantic+acoustic) or 12Hz 16-layer RVQ with semantic/acoustic disentanglement
Model backbone Custom transformer (from scratch) Qwen3 LM (pretrained text LLM) — benefits from language understanding, BPE text tokens, and instruction-following out of the box
Codebook 1 generation AR model with sampling Same principle, but backbone is full Qwen3 LM with much stronger context understanding

Languages English only 10 languages with cross-lingual voice cloning
Training data 60K hours (LibriLight) 5M+ hours multilingual (83× more)
Implementation Checklist
What you need to build / fine-tune Qwen3-TTS
  • 1.
    Tokenizer — Use the released Qwen/Qwen3-TTS-Tokenizer-12Hz (or 25Hz). These are frozen during LM training — treat them as black-box encode/decode modules, same as EnCodec in VALL-E.
  • 2.
    LM backbone — Initialize from a Qwen3 checkpoint (0.6B or 1.7B). Add a linear prediction head on top for cb0 token prediction. No special architectural changes needed.
  • 3.
    Speaker encoder — A learnable module that maps reference audio → embedding vector. Train jointly with the LM. During inference, prepend this embedding to the input sequence.
  • 4.
    MTP module (12Hz only) — A lightweight module that takes cb0 token sequence and predicts cb1–cb15 in one shot per frame. Analogous to VALL-E's NAR model but simpler (no separate transformer — just a prediction head).
  • 5.
    Code2Wav — For 12Hz: use the released causal ConvNet decoder. For 25Hz: DiT + BigVGAN (both are released). These are frozen waveform reconstructors — don't train them.
  • 6.
    Pre-training data — 5M+ hours is what they used. For fine-tuning, the CustomVoice / VoiceDesign variants use lightweight speaker adaptation on much smaller datasets.
  • 7.
    Training loss — Cross-entropy on cb0 token predictions (LM backbone). For MTP: cross-entropy on cb1–cb15 predictions given cb0. Optionally add DPO on top with human preference pairs.
  • 8.
    Inference — Autoregressive sampling on cb0 (greedy or temperature), then MTP for cb1–cb15 (single forward pass per frame), then Code2Wav decode. For streaming: emit audio packets as tokens arrive.
🚫 Do NOT use beam search for the LM backbone — same footgun as VALL-E. Token-level sampling or greedy decoding only. Beam search causes repetitive loops in autoregressive TTS.
⚠️ Ethical note: The 3-second voice cloning is extremely powerful. The paper acknowledges misuse risks (spoofing, impersonation) but provides no technical safeguards (no watermarking, no consent verification). Be responsible with deployment.
