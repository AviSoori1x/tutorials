01 — Motivation

What problem is this
paper trying to solve?

Reinforcement Learning with Verifiable Rewards (RLVR) works fantastically when the reward signal is unambiguous — math has one correct answer, code either passes tests or it doesn't. But real-world tasks don't work that way.

In medicine, a response might be numerically correct but clinically unsafe. In science explanation, an answer could be factually right but logically incomplete. A single scalar like "7/10" from a judge LLM is too coarse — it can be influenced by response length, formatting quirks, writing style, and annotator biases.

🎯 RLVR — Easy Cases

  • Math: is the final answer correct?
  • Code: do the test cases pass?
  • Single binary reward signal
  • No ambiguity in correctness
  • Scales without human annotation

🌐 Real-World Reasoning — Hard Cases

  • Medicine: safety + accuracy + completeness
  • Science: reasoning + correctness + clarity
  • Multi-dimensional quality judgments
  • No single ground truth
  • Subjective and objective criteria both matter
Key Tension

Preference-based reward models (RLHF) overfit superficial artifacts — response length, formatting, annotator biases — and require massive pairwise comparison datasets. There needed to be a middle ground between sparse binary correctness and noisy human preference scores.

02 — Core Idea

Rubrics as Rewards —
the one-sentence version

"Instead of rewarding an LLM with one vague score like 7/10, reward it using a checklist of what a good answer should do."

For each prompt, generate an instance-specific rubric — a small, targeted checklist of criteria tailored to that exact question. Then judge the model's answer against that checklist and turn the result into a scalar reward for RL training via GRPO.

Old Approach — Direct Likert

Reward = "This answer feels like a 7/10" — one vague scalar, influenced by length, polish, formatting.

New Approach — Rubrics as Rewards

Reward = "It satisfied 5 of the 7 important things this answer should contain." — structured, interpretable, task-specific.

Result

A middle ground between binary correctness (too simple) and human preference scores (too fuzzy), with rubrics as instance-specific, reusable reward functions.

03 — Formal Setup

The mathematical
formulation

The paper defines a structured reward function using instance-specific rubric criteria. Given an input prompt x and a sampled response ŷ ~ πθ(· | x) from a policy parameterized by θ:

Each prompt x is associated with a set of k rubric items {(w_j, c_j)}, where:

Notation

x = input prompt
ŷ = model response (sampled from policy)
k = number of rubric items for this prompt
w_j ∈ ℝ = weight of criterion j
c_j: (x, ŷ) → {0, 1} = binary correctness function

Each c_j asks: "Did the response satisfy criterion j given the prompt?" — returning 1 if yes, 0 if no.

Explicit Aggregation — Equation 1

r(x, ŷ) = Σⱼ wⱼ · cⱼ(x, ŷ) / Σⱼ wⱼ
Normalized weighted sum — comparable across prompts with differing rubric counts

Normalization ensures rewards are comparable across prompts that differ in rubric count or weights. The binary check c_j can be extended to continuous-valued scores, though binary is used in their experiments.

Implicit Aggregation — Equation 2

r_implicit(x, ŷ) = f_φ(x, ŷ, {dⱼ}ᵏⱼ₌₁)
Judge LLM f_φ takes prompt, response, and all rubric descriptions as input, outputs a holistic 1–10 scalar

Here f_φ is an LLM-based judge that computes a holistic reward score, avoiding the need to manually tune weights. The reward is normalized to [0, 1] during training.

04 — Aggregation Strategies

Explicit vs Implicit —
what's the real difference?

RaR-Explicit

  • Judge checks EACH item separately: pass/fail
  • You combine scores manually via weighted sum
  • Interpretable — you can inspect each score
  • You know exactly why a score happened
  • Brittle if manual weights are miscalibrated
  • Weights used: Essential=1.0, Important=0.7, Optional=0.3, Pitfall=0.9

RaR-Implicit ★ Best

  • Judge sees the WHOLE rubric at once
  • Outputs a single holistic 1–10 rating
  • No manual weight tuning required
  • Can capture interactions between criteria
  • Less transparent (judge reasoning is hidden)
  • Consistently outperforms explicit in experiments
Paper's Conclusion

RaR-Implicit attains the strongest results overall. The fixed weighted sums in RaR-Explicit offer more control but can be brittle. The authors view the choice as application-dependent — practitioners who need interpretability should prefer Explicit; those who want peak performance should use Implicit. Future work could explore learned or dynamic weighting strategies that maintain interpretability while improving adaptability.

05 — Theoretical Framing

How RaR subsumes
standard RLVR

Remark 1 (from the paper): The RLVR setting is a special case of rubric-based rewards defined in Equation 1, where k = 1, w₁ = 1, and c₁(x, ŷ) reduces to a single verifiable correctness function:

r_RLVR(x, ŷ) = match(y, ŷ)
where match(y, ŷ) ∈ {0, 1} — exact match or test case execution

Standard RLVR

1️⃣

k = 1 criterion
w = 1
Binary match

Rubrics as Rewards

🗂️

k = 7–20 criteria
Flexible weights
Subjective + objective

This neat conceptual point means the authors are not replacing RLVR — they're showing that RLVR is the trivial corner case of a much richer framework. Rubric-based rewards generalize RLVR by enabling multi-dimensional supervision, flexible weighting across criteria, and the incorporation of both objective and subjective aspects of response quality.

06 — Desiderata

What makes a
good rubric?

The paper identifies four desiderata for effective rubric generation:

🔬
Grounded in Expert Guidance
Rubrics should reflect domain expertise by capturing the essential facts, reasoning steps, and conclusions necessary for correctness. Ideally this grounding comes from human experts or high-quality proxies (like strong reference answers).
📐
Comprehensive Coverage
Rubrics should span multiple dimensions of response quality: factual accuracy, logical coherence, completeness, style, and safety. Negative criteria (pitfalls) help identify frequent or high-risk errors that undermine overall quality.
⚖️
Criterion Importance
Some dimensions matter more than others. Factual correctness must outweigh stylistic clarity. Assigning weights ensures this prioritization — whether through categorical tags, explicit numeric values, or learned schemes.
🔒
Self-Contained Evaluation
Each rubric item should be independently actionable. Human annotators or automated judges should be able to assess it in isolation without requiring external context or hidden domain-specific knowledge. This is a critical design requirement.
Why "Self-Contained" is Critical

A rubric item should not require the judge to implicitly know: "Well, obviously in cardiology you'd know this…" The rubric item itself must spell out exactly what to look for. Without self-containment, smaller judge models fail silently because they lack the domain knowledge to resolve the implicit context.

Rubric Category System

Category definitions and weight ranges

Essential
Critical facts or safety checks. If missing, the response is invalid. These are non-negotiable requirements.
w = 5
Important
Key reasoning, completeness, or clarity. Strongly affects quality but doesn't invalidate the response outright.
w = 3–4
Optional
Helpful style or extra depth. Nice to have but not deal-breaking. Examples: providing concrete analogies, additional context.
w = 1–2
Pitfall
Common mistakes or omissions specific to this prompt. Each Pitfall description begins "Does not mention…" or "Recommends…". Phrased positively in scoring (avoiding the pitfall = positive score).
w = −1 or −2

In experiments, numerical weights are mapped from categorical labels: Essential=1.0, Important=0.7, Optional=0.3, Pitfall=0.9 for the explicit aggregation variant.

Notable finding — generalization

Gains on GPQA-Diamond (a multiple-choice benchmark) from a model trained with rubric-based RL on free-form medicine/science questions shows that rubric-induced reasoning skills generalize beyond rubric-based evaluation formats. The model is learning something deeper, not just gaming rubric scoring.

Why RaR-Predefined Fails

The predefined method uses generic rubrics like "is concise", "contains correct information" for every prompt. It significantly underperforms — even worse than baseline on GPQA. Generic criteria miss prompt-specific requirements and common failure modes, producing misaligned reward signals. This confirms that instance-specific synthesis is not optional — it is the mechanism.

Rubrics that include a broader range of criteria (Essential + Important + Optional) outperform those limited to essential checks alone. Richer evaluation signals lead to better learning.
🏷️

Removing the categorical weight labels (Essential, Important, etc.) shows minimal performance difference. The specific weighting scheme matters less than simply having good prompt-specific coverage.
⚠️
Synthetic Pitfalls are Hard
Pitfall criteria show minimal gain when auto-generated, likely because generating effective pitfalls requires anticipating the model's specific failure modes — a task demanding human intuition and domain expertise that LLMs struggle to replicate synthetically.

12 — Concrete Example

The sodium bicarbonate
medical example

The paper includes a worked example from the RaR-Medicine dataset that illustrates perfectly why a single score is insufficient. This is a real training example:

Question (from RaR-Medicine)

A 50-year-old male patient weighs 65 kg with a pH of 7.05, PCO₂ of 15 mmHg, HCO₃ of 5 mEq/L, and a base deficit of −40 mEq/L. How much sodium bicarbonate should be administered in the first 4 hours to correct his metabolic acidosis?

Reference Answer

The formula: Bicarbonate required = Base Deficit (mEq/L) × Body Weight (kg) × 0.3

Applying: 40 × 65 × 0.3 = 780 mEq

However, standard practice is to administer only a partial correction. Administering about 150 mEq over the first 4 hours is a prudent approach, aligning with safe clinical guidelines to avoid rapid overcorrection.

Generated rubric (7 items with weights)

Essential
Bicarbonate Calculation (w=5): The response must correctly identify and apply the formula (Base Deficit × Body Weight × 0.3) to determine the bicarbonate requirement.
w = 5
Essential
Safe Dosing Recommendation (w=5): The response must state a clear recommendation of administering about 150 mEq of sodium bicarbonate over the first 4 hours.
w = 5
Important
Partial Correction Justification (w=4): The response should explain that only a partial correction is administered initially to avoid complications from rapid overcorrection.
w = 4
Important
Step-by-Step Calculation (w=3): The response must detail the calculation steps: 40 mEq/L × 65 kg × 0.3 = 780 mEq, before noting the adjusted dose for safety.
w = 3
Optional
Base Deficit Interpretation (w=2): The response may mention that the base deficit of −40 mEq/L indicates severe metabolic acidosis requiring cautious treatment.
w = 2
Important
Patient Data Accuracy (w=3): The response must accurately incorporate the patient's weight of 65 kg and his critical pH, PCO₂, and HCO₃ values into the explanation.
w = 3
Pitfall
Avoid Overcorrection Risk (w=−1): Does not mention the risks associated with rapid overcorrection of metabolic acidosis if only the full calculated bicarbonate amount is administered.
w = −1
Why this matters

A response could be numerically correct (780 mEq) but clinically dangerous if it recommends giving all of it in 4 hours. A rubric catches this — a vague 1–10 judge score might give it 7/10 because the math is right. The pitfall criterion specifically penalizes the failure to mention overcorrection risk, and the "Safe Dosing Recommendation" essential criterion requires the 150 mEq partial correction — two separate checks that a single Likert score cannot distinguish.

13 — Implementation

Code examples —
from toy to full loop

The explicit rubric aggregation formula (Equation 1) in plain Python:

# Toy example: explicit rubric reward (Equation 1)

rubric_items = [
    {"name": "mentions diagnosis", "weight": 5, "passed": 1},
    {"name": "explains key symptom", "weight": 4, "passed": 1},
    {"name": "mentions treatment", "weight": 3, "passed": 0},
    {"name": "avoids dangerous advice", "weight": 5, "passed": 1},
]

weighted_sum = sum(item["weight"] * item["passed"] for item in rubric_items)
total_weight = sum(item["weight"] for item in rubric_items)

reward = weighted_sum / total_weight
print("Reward:", reward)
# → Reward: 0.8235 (14/17)

PyTorch version — showing how rewards flow into an RL update via REINFORCE-style loss:

import torch
import torch.nn as nn
import torch.optim as optim

# ── Explicit rubric reward (vectorized) ──────────────
weights = torch.tensor([5.0, 4.0, 3.0, 5.0])
checks  = torch.tensor([1.0, 1.0, 0.0, 1.0])
reward = (weights * checks).sum() / weights.sum()
print("Rubric reward:", reward.item())  # → 0.8235

# ── Toy policy + REINFORCE loop ──────────────────────
policy    = nn.Linear(8, 3)            # 3 candidate responses
optimizer = optim.Adam(policy.parameters(), lr=1e-3)

x       = torch.randn(1, 8)            # fake prompt embedding
logits  = policy(x)
probs   = torch.softmax(logits, dim=-1)

dist    = torch.distributions.Categorical(probs)
action  = dist.sample()
logprob = dist.log_prob(action)

# reward from rubric evaluation (gpt-4o-mini in practice)
reward = torch.tensor(0.82)

loss = -logprob * reward       # REINFORCE loss
optimizer.zero_grad()
loss.backward()
optimizer.step()

More complete demo — computing rubric rewards for multiple candidate responses:

import torch

responses = [
    "The diagnosis is pneumonia. The fever and cough support this. Start antibiotics.",
    "It might be an infection.",
    "The diagnosis is pneumonia, but no treatment is needed.",
]

rubric = [
    {"name": "states diagnosis",           "weight": 5},
    {"name": "explains supporting evidence", "weight": 4},
    {"name": "mentions treatment",          "weight": 3},
    {"name": "avoids unsafe advice",        "weight": 5},
]

def check_response(response: str):
    r = response.lower()
    return torch.tensor([
        1.0 if "pneumonia"   in r else 0.0,
        1.0 if ("fever" in r or "cough" in r) else 0.0,
        1.0 if ("antibiotic" in r or "treatment" in r) else 0.0,
        0.0 if "no treatment is needed" in r else 1.0,
    ])

weights = torch.tensor([item["weight"] for item in rubric], dtype=torch.float32)

for i, response in enumerate(responses):
    checks = check_response(response)
    reward = (weights * checks).sum() / weights.sum()
    print(f"Response {i}: reward = {reward.item():.3f}")
# Response 0: reward = 1.000  (all criteria met)
# Response 1: reward = 0.294  (only diagnosis partial)
# Response 2: reward = 0.529  (correct dx, but "no treatment needed" triggers pitfall)

14 — Critical Assessment

Strengths
& where to be cautious

Where this paper is strongest

🔁
A. Reframes Reward Design
Not "learn a black-box reward model only," but "use structured, human-readable criteria." This connects reward shaping to natural language decomposition in a clean, interpretable way.
📐
B. Explains Why Smaller Judges Improve
The mechanism is clear: rubrics reduce ambiguity for the evaluator. A judge that doesn't need to holistically assess an answer, but just answer 8 binary questions, makes better use of limited model capacity.
🧮
C. Cleanly Generalizes RLVR
The formalization that RLVR = rubric with one criterion (k=1, w=1, binary match) is elegant and practically useful. It unifies verifiable and non-verifiable rewards under one framework.
🔀
D. Demonstrates Transfer
Training with rubric-based rewards on free-form medical text improves GPQA-Diamond multiple-choice accuracy. This suggests the model learns generalizable reasoning, not benchmark-specific surface hacks.

Caveats & limitations

Practical Risks

  • Rubric quality is everything — bad rubric in, bad reward out. The whole framework rests on rubric quality.
  • Generated rubrics encode generator biases — especially in subtle high-stakes domains like medicine.
  • Implicit aggregation is less interpretable — performs better but the judge does hidden reasoning again.
  • Synthetic pitfalls are weak — ablations show pitfall criteria don't help much when auto-generated without human intuition.

Scope & Future Work

  • Only validated on medicine and science — authors explicitly flag dialogue, tool use, agentic tasks as future work.
  • Uses off-the-shelf LLM judges — specialized evaluators or generative reward models may yield further gains.
  • Only two aggregation strategies explored — learned continuous weights or curriculum-based dynamic weighting is open.
  • No analysis of training stability, reward variance across checkpoints, or catastrophic forgetting.
