0The big picture

The big picture

A transformer looks at a sequence — words, image patches, audio chunks — and lets every element attend to every other. That mechanism is attention.

But attention has a fundamental flaw: by default, it does not know order. Shuffle the input and the outputs just shuffle the same way. Position has to be injected separately.

The problem ↗ — Attention only sees vectors. If you swap "cat" and "sat", the model sees the same set of vectors in a different order and produces the same results. Order is invisible to vanilla attention.
Every token can look at every other. But there's no notion of who came first.
1What is a vector?

What is a vector?

A vector is an ordered list of numbers. That's it. In ML, almost everything becomes a vector at some point.

x = torch.tensor([2.0, -1.0, 0.5])   # a 3-dimensional vector
Three ways to picture a vector
1 — list of numbers
[ 2.0, −1.0, 0.5 ]
Each slot = one feature
2 — point in space
x=2.0, y=−1.0, z=0.5
Coordinates in N-dim space
3 — direction + length
↗ magnitude 2.3
Arrow from origin to point
All three pictures are useful. The rotation picture (3) is the one RoPE lives in.

What is an embedding?

An embedding maps discrete tokens (words, characters, patch IDs) to vectors. The key point: the embedding captures what the token means — not where it appears.

Sentence A

Same tokens. Same embedding vectors. Completely different meaning. Order is everything. A model that can't see order will mix these up.

Each score in QKᵀ is a dot product: how much does this query match this key? Softmax turns scores into a probability distribution. Then we use those weights to mix the values.

weights = torch.softmax(scores, dim=-1)  # [seq, seq] — rows sum to 1
out     = weights @ V                     # [seq, d_v] — weighted mix

7Sinusoidal PE

Sinusoidal PE: what it actually does

The original Transformer built position vectors from alternating sine and cosine waves at many frequencies:

PE(pos, 2i) = sin(pos / 10000^(2i/d))

PE(pos, 2i+1) = cos(pos / 10000^(2i/d))

Structure of a sinusoidal position vector
sin(ω₀·p)
cos(ω₀·p)
sin(ω₁·p)
cos(ω₁·p)
sin(ω₂·p)
cos(ω₂·p)
···
sin(ω_{d/2}·p)
cos(ω_{d/2}·p)
Pair 0 (ω₀ = 1) — changes the fastest. Differentiates nearby positions.
Pair d/2 (ω_{d/2} ≈ 0.0001) — barely changes. Provides long-range structure.
Every position gets a unique fingerprint vector, built from multi-frequency oscillations.
8Why sin AND cos?

Why sine AND cosine alternate

Sine alone is ambiguous — different angles can share the same sine value. But the pair [sin(θ), cos(θ)] is a unique point on the unit circle, identifying θ exactly.

sin alone is ambiguous — (sin, cos) is not
❌ sin(θ) alone
sin(30°) = 0.5
sin(150°) = 0.5
Same value — two different positions. Can't tell which is which.
✓ (sin θ, cos θ) — unique point on unit circle
30°
150°
Green dot ≠ purple dot.
Different positions, different points.
That's why dimensions always come in (sin, cos) pairs. The pair gives you the angle unambiguously.
9Why different frequencies?

Why different frequencies are used

A single sine/cosine pair would repeat — two different positions would get the same encoding. Multiple frequencies create a unique fingerprint for every position, like a clock with second, minute, and hour hands.

High frequency (early pairs) — changes fast, encodes fine-grained local position.
Low frequency (later pairs) — changes slowly, encodes coarse long-range position.

Frequency bands — how fast each pair changes with position
pair 0 (fast)
ω₀ = 1.000
pair 2
ω₂ = 0.046
pair 4
ω₄ = 0.002
pair 7 (slow)
ω₇ ≈ 0.0001
Fast pairs tell the model about nearby tokens. Slow pairs tell it about the token's place in the whole sequence.
10Build sinusoidal PE

Build sinusoidal PE in PyTorch

Readable version — nested loops make structure clear
def sinusoidal_pe_slow(seq_len, d_model):
    P = torch.zeros(seq_len, d_model)
    for pos in range(seq_len):
        for i in range(d_model // 2):
            omega = 1 / (10000 ** (2*i / d_model))
            P[pos, 2*i]   = math.sin(omega * pos)
            P[pos, 2*i+1] = math.cos(omega * pos)
    return P
Fast vectorized version — use in practice
def sinusoidal_pe(seq_len, d_model):
    pos = torch.arange(seq_len).unsqueeze(1)
    i   = torch.arange(0, d_model, 2).float()
    div = torch.exp(
        -math.log(10000.0) * i / d_model
    )
    angles = pos * div
    P = torch.zeros(seq_len, d_model)
    P[:, 0::2] = torch.sin(angles)
    P[:, 1::2] = torch.cos(angles)
    return P
11Absolute vs relative

Absolute vs relative positional encoding

Sinusoidal PE is absolute — position 0 gets one specific vector, position 1 another, etc.

But it's designed so that relative shifts are linearly recoverable — if you know PE(p), you can compute PE(p+k) with a linear transformation. This is the rotation property.

Absolute vs relative: what this means
Absolute positional encoding
Each position gets its own fixed vector
p=0
→ [ sin(0), cos(0), sin(0), cos(0), … ]
p=1
→ [ sin(1), cos(1), sin(ω₁), cos(ω₁), … ]
p=7
→ [ sin(7), cos(7), sin(7ω₁), … ]
Relative positional encoding (goal)
Dot products depend on offset, not absolute position
"token at p=3 and token at p=7" → offset = 4

Whether it's positions (3,7) or (100,104), the attention score should be the same.
Sinusoidal PE is absolute, but architecturally structured so relative distances are preserved. RoPE goes further and makes attention scores directly relative.
12Shift = rotation

The rotation idea behind sinusoidal PE

For one frequency ω, the positional pair is [ sin(ωp), cos(ωp) ]. Moving from position p to p+k is a rotation of that 2D pair by angle ωk:

PE(p+k) = R(ωk) @ PE(p)

Shift in position = rotation of the (sin, cos) pair
pos p
pos p+k
Rotation matrix
R(θ) = | cos θ −sin θ |
        | sin θ cos θ |
Key property
PE(p+k) = R(ω·k) @ PE(p)
Shift in position = fixed rotation. So relative offsets are linear transformations of the encoding.
This is the deep trick in sinusoidal PE. But note: Q and K are mixed with semantics first, then rotated. RoPE improves on this.
def shift_matrix(k, omega):
    # Moving k positions = rotating the (sin,cos) pair by angle omega*k
    return torch.tensor([
        [ math.cos(omega*k),  math.sin(omega*k)],
        [-math.sin(omega*k),  math.cos(omega*k)]
    ])

# Verify: PE(p+k) = shift_matrix(k) @ PE(p)  ✓
13Why additive PE isn't perfect

Why additive PE is not perfect

The problem with adding position to the input
token embedding

The issue: semantics and position get blended inside Q and K. The projection matrix W_Q can't cleanly separate "what this token means" from "where it is." Position information leaks into the semantic space — and semantic information leaks into the position signal.
It works, but it's architecturally messy. RoPE solves this by injecting position directly into attention scores, after the semantic projection.
14RoPE from first principles

RoPE from first principles

Key idea: don't add position to the input. Instead, rotate Q and K according to their position — after the semantic projection.

This way, position only affects the attention score (Q·K dot products), not the V projection and not the semantic embedding itself.

Sinusoidal PE vs RoPE — where position enters
Additive sinusoidal PE
① X + P[p]  ← position added here
↓ projection

K = (X+P) @ W_K
↓
scores = Q @ Kᵀ
Position and semantics mixed before projection
RoPE
Q = X @ W_Q
K = X @ W_K
↓ then rotate
Q_rot = rotate(Q, p)  ← position here
K_rot = rotate(K, p)  ← position here
↓
scores = Q_rot @ K_rotᵀ
Semantics first, position injected into scores directly
15Why rotation fits attention

Why rotation is a perfect fit for attention

The attention score between token p and token m is a dot product: q_p · k_m = |q||k| cos(angle between them).

Rotation changes the angle but preserves the norm. So rotating Q and K by their positions changes their angle relationship in a structured way, encoding position directly into the score — without distorting the magnitudes that carry semantic information.

What rotation does to dot products
Before rotation
q·k = |q||k| cos(θ_semantic)

Score = how semantically similar these two tokens are
After RoPE
R(ωp)·q · R(ωm)·k
= |q||k| cos(θ_semantic + ω(p−m))

Score = semantic similarity modulated by relative distance p−m
The magic: the relative position (p−m) appears as an additive term in the angle, so the dot product naturally depends on how far apart the tokens are.
The norm is preserved. Only the angle changes — and angle is exactly what dot products measure.
162D rotation refresher

2D rotation refresher

Rotating a 2D vector by angle θ
Rotation matrix R(θ)
⎡ cos θ   −sin θ ⎤
⎣ sin θ    cos θ ⎦
Norm-preserving: |R(θ)v| = |v| always
Composable: R(θ₁) · R(θ₂) = R(θ₁+θ₂)
Invertible: R(θ)⁻¹ = R(−θ)
def rotate_2d(v, theta):
    cos_t, sin_t = math.cos(theta), math.sin(theta)
    R = torch.tensor([
        [cos_t, -sin_t],
        [sin_t,  cos_t]
    ])
    return R @ v
17RoPE in one 2D pair

RoPE in one 2D pair

Take the first two dimensions of Q: [q₀, q₁]. At position p, RoPE rotates this pair by angle ωp. Same for K at position m: rotate by ωm.

The dot product becomes: R(ωp)·q · R(ωm)·k = q · R(ω(m−p)) · k

The score only depends on (m−p) — the relative distance.

Query at position p, Key at position m — in one 2D pair
R(ωp)·q
R(ωm)·k
Dot product proof
R(ωp)·q  ·  R(ωm)·k
= q  ·  R(ωp)ᵀ R(ωm)  ·  k
= q  ·  R(ω(m−p))  ·  k
= depends only on (m−p)
This is the core proof. The absolute positions p and m disappear — only the difference survives in the score.
18RoPE in many dimensions

RoPE in many dimensions

Real embeddings are hundreds of dimensions, not 2. RoPE handles this by splitting the embedding into consecutive 2D pairs, and rotating each pair with a different frequency ω_i:

Splitting a d=8 vector into 4 independent rotation planes
q₀, q₁
q₂, q₃
q₄, q₅
q₆, q₇
rotate by ω₀·p
(fastest)
rotate by ω₁·p
rotate by ω₂·p

(slowest)
q₀', q₁'
q₂', q₃'

Each pair is an independent rotation plane. Different frequencies ω_i = 1/base^(2i/d) — same formula as sinusoidal PE, same multi-scale idea. Each pair encodes relative distance at a different granularity.
d/2 pairs, d/2 rotation planes, d/2 frequency bands. Each pair contributes one "view" of relative position to the dot product.
19PyTorch 1D RoPE

Clean PyTorch implementation of 1D RoPE

def apply_rope(x, base=10000):
    """
    x: [seq_len, dim]   (one attention head)
    Returns x with each 2D pair rotated by its position-dependent angle.
    """
    seq_len, dim = x.shape
    assert dim % 2 == 0

    # Build angles: shape [seq_len, dim//2]
    positions = torch.arange(seq_len).float().unsqueeze(1)    # [seq, 1]
    pair_idx  = torch.arange(dim // 2).float().unsqueeze(0)  # [1, dim//2]
    inv_freq  = 1.0 / (base ** (2 * pair_idx / dim))
    angles    = positions * inv_freq                               # [seq, dim//2]

    cos = torch.cos(angles)   # [seq, dim//2]
    sin = torch.sin(angles)   # [seq, dim//2]

    # Split into even and odd dimensions (the two halves of each pair)
    x_even = x[:, 0::2]   # [seq, dim//2]  — first element of each pair
    x_odd  = x[:, 1::2]   # [seq, dim//2]  — second element of each pair

    # Apply rotation: [x', y'] = [x·cos − y·sin, x·sin + y·cos]
    out = torch.empty_like(x)
    out[:, 0::2] = x_even * cos - x_odd * sin
    out[:, 1::2] = x_even * sin + x_odd * cos

    return out
Line by line: build angles[pos, pair] = pos / base^(2·pair/d). Take even/odd elements as the two components of each 2D pair. Apply the 2D rotation formula. That's it.
20Attention with RoPE

Minimal self-attention with RoPE

def self_attention_rope(X, W_Q, W_K, W_V):
    Q = X @ W_Q   # semantic projection first
    K = X @ W_K
    V = X @ W_V

    Q = apply_rope(Q)   # then rotate by position
    K = apply_rope(K)   # V is NOT rotated — position only affects scores

    scale   = math.sqrt(Q.shape[-1])
    scores  = Q @ K.T / scale          # scores depend on relative position
    weights = torch.softmax(scores, dim=-1)
    out     = weights @ V

    return out
Note: V is not rotated — only Q and K. Position should only affect which tokens attend to which (the scores), not the content that gets aggregated (the values).

2D RoPE for image patches

In text, a token has one coordinate: p. In images, a patch has two: (x, y). We allocate half the rotation planes to x and half to y, then rotate each group with its corresponding coordinate.

Used in ViT-style models and multimodal LLMs with image tokens.
def rope_2d_single(vec, x_pos, y_pos, base=10000):
    """vec: [dim] — one token's Q or K vector, dim must be divisible by 4"""
    d = vec.shape[0]
    half = d // 2

    # First half: rotate using x coordinate
    vec_x = apply_rope(vec[:half].unsqueeze(0).unsqueeze(0), base=base)
    # Second half: rotate using y coordinate
    vec_y = apply_rope(vec[half:].unsqueeze(0).unsqueeze(0), base=base)

    return torch.cat([vec_x.squeeze(), vec_y.squeeze()], dim=0)

29Common confusions

Common confusions — cleared up

Are the sin/cos values literally added to the word vector?
Yes — in original sinusoidal PE, they are added elementwise. X_with_pos = X + P where P is the sine/cosine matrix.
Do sin and cos really alternate dimension by dimension?
Yes — dim 0 = sin(ω₀·p), dim 1 = cos(ω₀·p), dim 2 = sin(ω₁·p), dim 3 = cos(ω₁·p), etc.
Is QKᵀ a dot product or a matrix multiply?
Both. It's a matrix multiply overall [seq × seq], but every single entry is the dot product between one query vector and one key vector.
Is sinusoidal PE absolute or relative?
It's absolute — each position gets its own fixed vector. But it's constructed so relative shifts are recoverable via linear transformation. Not the same as truly relative PE.
Is RoPE absolute or relative?
It uses absolute positions to compute rotation angles, but the attention dot products only depend on relative offset (p−m). Behaviorally relative, mechanistically absolute.
Why does rotating preserve the norm?
Rotation is an orthogonal transformation: R(θ)ᵀR(θ) = I. So |R(θ)v|² = vᵀR(θ)ᵀR(θ)v = vᵀv = |v|². The length never changes, only the direction.
Why isn't V rotated in RoPE?
V is the content that gets aggregated. Position should determine how much each token contributes (via scores), not what it contributes. Rotating V would corrupt the value information.
Does the base (10000 or 1M) matter?
Yes. Larger base = slower-rotating frequencies = longer context before positions repeat or alias. LLaMA uses 10,000. Mistral and Qwen use 1,000,000 for longer contexts.
30Final compressed intuition

Final compressed intuition

1. Attention doesn't see order. You have to inject it.
2. Sinusoidal PE adds multi-frequency sin/cos vectors to token embeddings. Each position gets a unique fingerprint. Relative shifts = linear rotation of the encoding. Works, but mixes position and semantics.
3. RoPE is cleaner: compute Q and K first (pure semantics), then rotate each by its position. The dot product then naturally encodes relative distance — by construction, not by training. No position vector is added anywhere.
4. For 2D/3D data: split embedding pairs into groups, one per axis. Rotate each group with its axis coordinate. Relative offsets in each axis appear in the score independently.
The one equation that makes RoPE work
R(ω·p)·q  ·  R(ω·m)·k
=   q  ·  R(ω·(m−p))  ·  k
=   f( m−p  only )
The absolute positions p and m cancel. Only the relative distance (m−p) survives.

def apply_rope(x, base=10000):
    seq_len, dim = x.shape
    positions = torch.arange(seq_len).float().unsqueeze(1)
    pair_idx  = torch.arange(dim//2).float().unsqueeze(0)
    inv_freq  = 1.0 / (base ** (2*pair_idx/dim))
    angles    = positions * inv_freq
    cos, sin  = torch.cos(angles), torch.sin(angles)
    x_e, x_o  = x[:, 0::2], x[:, 1::2]
    out = torch.empty_like(x)
    out[:, 0::2] = x_e*cos - x_o*sin
    out[:, 1::2] = x_e*sin + x_o*cos
    return out

def attention_rope(X, W_Q, W_K, W_V):
    Q = apply_rope(X @ W_Q)
    K = apply_rope(X @ W_K)
    V = X @ W_V  # no rope on V!
    s = Q @ K.T / math.sqrt(Q.shape[-1])
    w = torch.softmax(s, dim=-1)
    return w @ V
