🤔 What Problem Does RT-2 Solve?

Modern Vision-Language Models (VLMs) have read most of the internet. They know that rocks are hard and can be used as hammers. They know energy drinks help tired people. They know what strawberries are and which bowl fruit belongs in.

But a robot arm doesn't care about "knowing" things. A robot needs to produce exact numbers — how many millimeters to move its gripper, how many degrees to rotate its wrist. How do you bridge that gap?

The Gap

VLMs output text. Robots need motor commands — continuous 6-dimensional vectors describing end-effector position and rotation. These seem completely incompatible. Every previous approach used a separate low-level controller to translate between them, which means the VLM's rich world knowledge never directly influences the physical motion.

What was tried before RT-2?

Approach: VLM as Planner (e.g. PaLM-E, SayCan)

VLM generates high-level instructions in text ("pick up the red block"), then a completely separate low-level policy executes them. The VLM never sees nor influences the actual motor commands. Its knowledge is "upstairs" and never reaches the motors.

Approach: Separate pre-trained vision encoder

Use CLIP or another VLM just to compute a visual embedding, feed it into a small policy network. The VLM's language knowledge is mostly lost — you're only borrowing its vision encoder, not its reasoning.

RT-2's Radical Answer

What if we just write robot actions as text? Then the VLM can output actions the exact same way it outputs words — no separate controller, no translation step, no knowledge barrier. The VLM directly generates the numbers that move the robot.

💡 The Big Idea in One Picture

Traditional approach — two separate systems: ──────────────────────────────────────────────────────────────── [Image + "pick up the apple"] → VLM → "move towards apple" ← language only ↓ [Separate policy network] ↓ [0.05, -0.12, 0.3, 5°, 2°, 8°] ← actions RT-2 approach — ONE unified system: ──────────────────────────────────────────────────────────────── [Image + "pick up the apple"] → VLM → "1 128 91 241 5 101 127" ↓ de-tokenize these 8 numbers ↓ [0.05, -0.12, 0.3, 5°, 2°, 8°, gripper, stop] The VLM IS the robot policy. One model, end-to-end. Same weights that know "rocks are hard" also move the gripper.
Why does this matter so much?

When the model weights are shared across language, vision, and actions, the robot gets all of the VLM's world knowledge for free. It can pick up a rock when asked to "find something to hammer a nail" not because we programmed that rule — but because the model has read thousands of web pages about hammers and rocks, and that knowledge now also controls the arm.

🎮 Concept 1: Representing Robot Actions as Text Tokens

This is RT-2's most important contribution. Let's understand it deeply.

What is the robot's action space?

The robot is a 7-DoF mobile manipulator. At each time step it outputs:

ROBOT ACTION — 8 numbers per timestep
terminatestop? (0/1)
Δxpos x
Δypos y
Δzpos z
Δrxrot x
Δryrot y
Δrzrot z
gripgripper
6-DoF end-effector delta (change in position + rotation) + gripper extension + termination signal

Step 1: Discretize the continuous values into 256 bins

Continuous numbers like 0.047 can't be directly fed as text tokens. So we split the range of each dimension into 256 equal buckets, and each value becomes a bin number (an integer 0–255).

PYTORCH
import torch
import numpy as np

def discretize_action(action_vector, num_bins=256, min_val=-1.0, max_val=1.0):
    """
    Convert a continuous robot action to discrete bin indices.

    action_vector: numpy array of shape (7,)
      [delta_x, delta_y, delta_z, delta_rx, delta_ry, delta_rz, gripper]
    Returns: list of 7 integers, each in [0, 255]
    """
    # Clip to valid range first
    action_vector = np.clip(action_vector, min_val, max_val)

    # Map [-1.0, 1.0] → [0, 255] uniformly
    # Think of it like: which of 256 equally-spaced buckets does this value fall into?
    bins = ((action_vector - min_val) / (max_val - min_val) * (num_bins - 1)).astype(int)
    return bins.tolist()

def undiscretize_action(bin_indices, num_bins=256, min_val=-1.0, max_val=1.0):
    """Reverse: convert bin indices back to continuous action values."""
    bins = np.array(bin_indices, dtype=np.float32)
    return bins / (num_bins - 1) * (max_val - min_val) + min_val

#                close gripper slightly
raw_action = np.array([0.05, -0.12, 0.03, 0.05, 0.02, 0.08, 0.3])

bins = discretize_action(raw_action)

#   (slightly different due to rounding — 256 bins gives ~0.008 resolution)

Step 2: Turn the bin numbers into a text string

Now each action is just 8 integers. Write them as a space-separated string — exactly like text! Add a terminate flag at the front (1=continue, 0=stop).

PYTHON
def action_to_string(terminate_flag, bin_indices):
    """
    Convert robot action to the text string format used by RT-2.

    Format: "terminate Δx Δy Δz Δrx Δry Δrz gripper"
    Example output: "1 128 91 241 5 101 127 200"
    """
    parts = [str(terminate_flag)] + [str(b) for b in bin_indices]
    return " ".join(parts)

def string_to_action(action_str):
    """Parse the model's output string back into robot commands."""
    parts = [int(x) for x in action_str.strip().split()]
    terminate = parts[0]    # 0 = keep going, 1 = stop
    bins      = parts[1:]   # 7 bin indices for the 7 action dimensions
    action    = undiscretize_action(bins)
    return terminate, action

# ── Full round trip ──────────────────────────────────────────────────
raw_action = np.array([0.05, -0.12, 0.03, 0.05, 0.02, 0.08, 0.3])
bins       = discretize_action(raw_action)

# This is what RT-2 is TRAINED TO PREDICT as output text:
action_str = action_to_string(terminate_flag=1, bin_indices=bins)
print(f"Model output string: '{action_str}'")
# → '1 134 113 130 134 131 138 165'

# And how the robot decodes that string to move:
terminate, decoded = string_to_action(action_str)
print(f"Terminate: {terminate}, Action: {np.round(decoded, 3)}")
# → Terminate: 1, Action: [ 0.051 -0.118  0.031  0.051  0.02   0.082  0.3 ]
Which tokens does RT-2 use for actions?

RT-2 fine-tunes two VLMs with different tokenizers:
• PaLI-X: integers 0–999 each have their own unique token, so action bins 0–255 map directly to those number tokens.
• PaLM-E: no such clean integer tokens exist, so RT-2 overwrites the 256 least-frequently used tokens in the vocabulary and reassigns them as action tokens. This is called symbol tuning and works surprisingly well.

🏗️ Concept 2: The RT-2 Architecture

RT-2 doesn't design a new architecture from scratch. Instead, it takes a pre-trained VLM and fine-tunes it to output action tokens. Two VLMs were used:

RT-2-PaLI-X (5B and 55B)

PaLI-X is a joint vision-language model from Google. It uses a large ViT image encoder + a large language model. Tested at 5B and 55B parameter scales. Integers 0–999 are unique tokens, making action encoding clean.

RT-2-PaLM-E (12B)

The same PaLM-E from the previous paper! Here it's used as the backbone for an RT-2 variant. The 256 rarest vocabulary tokens are repurposed as action tokens (symbol tuning).

RT-2 Architecture (same for both PaLI-X and PaLM-E backbones): ┌─────────────────────────────────────────────────────────────────┐ │ INPUTS at time t │ └─────────────────────────────────────────────────────────────────┘ │ │ ▼ ▼ [Camera Image] [Task Instruction text] "wrist + overhead cam" "pick up the apple" │ │ ▼ │ ┌─────────────┐ │ │ ViT Image │ │ │ Encoder │ │ │ (frozen or │ │ │ fine-tuned)│ │ └─────────────┘ │ │ image patch embeddings │ text token embeddings └──────────────┬────────────────────┘ ▼ ┌─────────────────────────────────────────────────────────────────┐ │ Large Language Model (decoder-only transformer) │ │ 5B / 12B / 55B parameters │ │ │ │ Processes the mixed sequence of image patches + text tokens │ │ using self-attention — same as in PaLM-E │ └─────────────────────────────────────────────────────────────────┘ │ ▼ auto-regressive output ┌──────────────────────────────────────┐ │ "1 128 91 241 5 101 127" │ ← 7 tokens predicted one by one └──────────────────────────────────────┘ │ ▼ de-tokenize + undiscretize ┌──────────────────────────────────────────────────────────────────┐ │ terminate=1 Δpos=[0.05, -0.12, 0.03] Δrot=[5°, 2°, 8°] │ │ gripper=0.3 │ └──────────────────────────────────────────────────────────────────┘ │ ▼ Robot arm executes this command New image captured → loop repeats at ~1-3 Hz ┌─────────────────────────────────────────────────────────────────┐ │ OUTPUT CONSTRAINT during inference: │ │ When RT-2 is in "robot mode", we only allow it to sample from │ │ the 256 action tokens (+ terminate). This prevents nonsense │ │ outputs like "the" or "apple" when we need numbers. │ │ On VQA tasks, the full vocabulary is allowed. │ └─────────────────────────────────────────────────────────────────┘
Output Constraint — an important engineering detail

During robot execution, at each decoding step RT-2 is only allowed to sample from the 8 action token types (256 bins × 7 dimensions + terminate). This is like a forced multiple-choice where the model can only pick from valid action bin numbers. It prevents the model from saying "I'll pick up the apple" when you need it to say "128" (a specific gripper position).

🔀 Concept 3: Co-Fine-Tuning (The Secret Sauce)

The most important training decision in RT-2 isn't the architecture — it's how you mix the data.

Three training strategies compared

A

Train from scratch on robot data only

Start with random weights. Train only on robot trajectory data. Result: terrible. Even a 5B model trained this way fails badly. There is simply not enough robot data (tens of thousands of trajectories) to train billions of parameters from scratch.

B

Fine-tune pre-trained VLM on robot data only

Start from a pre-trained VLM (billions of parameters, trained on internet-scale data). Then fine-tune only on robot data. Better than scratch, but catastrophic forgetting kicks in — the model forgets the web knowledge that makes it generalizable, and overfits to the robot distribution.

C

Co-fine-tune: robot data + original web VQA data simultaneously (RT-2)

Mix robot trajectories with the original VQA/captioning web data in every training batch. The model sees both at the same time. It can't forget web knowledge because it's still being trained on it. This is what RT-2 does, and it gives the best generalization by far.

PYTORCH — Co-fine-tuning data loader
from torch.utils.data import Dataset, DataLoader, WeightedRandomSampler
import torch

class RobotTrajectoryDataset(Dataset):
    """Robot demonstration dataset.
    Each item: (image, task_instruction, action_string)
    e.g.: (camera_frame, "pick up the apple", "1 128 91 241 5 101 127")
    """
    def __init__(self, size=10000):
        self.size = size

    def __len__(self): return self.size

    def __getitem__(self, idx):
        # In reality: load camera images and recorded joint positions
        # Here: fake tensors for illustration
        image   = torch.randn(3, 224, 224)
        prompt  = "Q: what action should the robot take to pick up the apple? A:"
        target  = "1 128 91 241 5 101 127"  # action tokens as text
        return {'image': image, 'prompt': prompt, 'target': target, 'type': 'robot'}


class WebVQADataset(Dataset):
    """Internet-scale VQA / captioning dataset.
    Same format, but targets are natural language answers.
    """
    def __init__(self, size=10_000_000):
        self.size = size

    def __len__(self): return self.size

    def __getitem__(self, idx):
        image  = torch.randn(3, 224, 224)
        prompt = "Q: What is in the image? A:"
        target = "a red apple on a white plate"  # natural language answer
        return {'image': image, 'prompt': prompt, 'target': target, 'type': 'vqa'}


# ── Co-fine-tuning: sample BOTH datasets in every batch ───────────────
#
# The trick: increase the sampling weight on robot data so it shows up
# often enough in each batch (since there is much less robot data).
#
# Think of it like: you have 10,000 robot examples and 10,000,000 web
# examples. Without reweighting, you'd almost never see robot data!

robot_dataset = RobotTrajectoryDataset(size=10_000)
web_dataset   = WebVQADataset(size=10_000_000)

# Assign weights: robot data gets upsampled heavily
# (RT-2 paper: ~50% of each batch is robot data)
robot_weight = 0.5 / len(robot_dataset)    # high weight per sample
web_weight   = 0.5 / len(web_dataset)      # low weight per sample (many more samples)

all_weights = ([robot_weight] * len(robot_dataset) +
               [web_weight]   * len(web_dataset))

# WeightedRandomSampler draws samples according to these weights
sampler = WeightedRandomSampler(
    weights=all_weights,
    num_samples=len(all_weights),
    replacement=True
)

# Combine both datasets
combined = torch.utils.data.ConcatDataset([robot_dataset, web_dataset])

# This DataLoader draws ~50% robot, ~50% web in every batch
loader = DataLoader(combined, batch_size=32, sampler=sampler)

# ── Training loop (shared weights, different targets!) ────────────────
for batch in loader:
    images   = batch['image']    # (32, 3, 224, 224)
    prompts  = batch['prompt']   # list of 32 text strings
    targets  = batch['target']   # list of 32 target strings
    types    = batch['type']     # 'robot' or 'vqa' for logging

    # The model sees both types in the SAME batch.
    # For 'robot' items → targets are action token strings like "1 128 91 241..."
    # For 'vqa' items   → targets are text strings like "a red apple..."
    # SAME model, SAME cross-entropy loss, SAME optimizer step.
    # This is the entire co-fine-tuning trick.
    break  # just showing structure

print("Co-fine-tuning: robot actions and web VQA share the same training loop!")
Why does co-fine-tuning work better?

Imagine you're training to be a chef AND a surgeon simultaneously. If you only practice surgery and stop cooking entirely, you'll forget how to cook. But if you alternate between cooking and surgery, you get good at both — and skills like "precise hand movements" transfer between them. Co-fine-tuning does this for the model: it stays fluent in "web knowledge" (VQA) while simultaneously learning "motor commands" (robot actions), and the two reinforce each other.

🧠 Concept 4: Chain-of-Thought for Robots

Standard language models can be prompted to "think step by step" before giving an answer. This dramatically improves reasoning. RT-2 applies the same idea to robot control — and it's a beautiful result.

Normal RT-2 output:

Instruction: "I'm hungry." VLM output: "1 128 124 136 121 158 111 255" ↓ robot picks up the rxbar chocolate bar

Chain-of-Thought RT-2 output:

Instruction: "I need to hammer a nail. What should I use?" VLM output: "Plan: pick up the rock. Action: 1 129 138 122 132 135 106 127" ↑─────────────────────┘ ↑────────────────────────────┘ Natural language reasoning Motor commands The model FIRST reasons in English, THEN outputs the action. It figured out "rocks can be used as improvised hammers" from web knowledge. That reasoning was never in any robot training example!
Why is this so powerful?

The chain-of-thought step happens inside the model's context — the plan text is output first, then the action tokens are generated conditioned on that plan. This means the model's reasoning about WHAT to do (informed by internet knowledge) directly shapes HOW it moves (the action tokens). The language and the motion are finally connected in one continuous flow.

PYTORCH — Chain-of-thought output format
# Normal RT-2: just action tokens
# ─────────────────────────────────────────────────────────────────────
# Input:  [image] + "Q: what action should robot take to pick apple? A:"
# Output: "1 128 91 241 5 101 127"

# Chain-of-Thought RT-2: plan THEN action tokens
# ─────────────────────────────────────────────────────────────────────
# Input:  [image] + "Instruction: I need to hammer a nail."
# Output: "Plan: pick up the rock. Action: 1 129 138 122 132 135 106 127"
#
#          └────────────────────┘          └─────────────────────────┘
#          plain English reasoning         8 action bin tokens

import torch

def parse_cot_output(model_output_str):
    """
    Parse chain-of-thought output that contains both a plan
    and action tokens.

    Example input:
      "Plan: pick up the rock. Action: 1 129 138 122 132 135 106 127"

    Returns: (plan_text, action_tokens)
    """
    if "Action:" in model_output_str:
        parts  = model_output_str.split("Action:")
        plan   = parts[0].replace("Plan:", "").strip()
        action = parts[1].strip()
    else:
        plan   = ""
        action = model_output_str.strip()

    return plan, action

# ── Examples from the paper ────────────────────────────────────────────
examples = [
    "Plan: pick up the rock. Action: 1 129 138 122 132 135 106 127",
    "Plan: pick 7up can. Action: 1 143 129 123 145 114 115 127",
    "Plan: move green can near green rice chip bag. Action: 1 130 129 121 131 127 128 127",
    "Plan: pick rxbar chocolate. Action: 1 128 129 125 131 125 128 127",
]

print("Chain-of-Thought RT-2 outputs:")
print("─" * 60)
for ex in examples:
    plan, action = parse_cot_output(ex)
    print(f"PLAN:   {plan}")
    terminate, motion = string_to_action(action)
    print(f"ACTION: terminate={terminate}, motion={[round(x,3) for x in motion]}")
    print()

How is the chain-of-thought trained?

For a small subset of robot training examples, the authors augment the target string to include a "Plan:" step before the action tokens. The model is fine-tuned for just a few hundred gradient steps on this augmented data. That's enough to unlock the capability — because the ability to reason in English was already there from VLM pretraining.

Data augmentation for chain-of-thought

Original target: "1 128 124 136 121 158 111 255"
Augmented target: "Plan: pick rxbar chocolate. Action: 1 128 124 136 121 158 111 255"

The "Plan:" text is automatically generated by prompting the base VLM to describe what the action achieves, given the original instruction. No human annotation needed for the plan text.

rt2_demo.py — Complete runnable code
"""
RT-2 Concept Demo
=================
Demonstrates all RT-2 core ideas in runnable PyTorch.

Key ideas shown:
 1. Action discretization into 256 bins
 2. Action-as-text-token format
 3. Simplified VLA model (VLM backbone + action output head)
 4. Co-fine-tuning data mixing strategy
 5. Output constraint (only valid action tokens at inference)
 6. Chain-of-thought decoding
 7. Closed-loop robot control simulation

Run with: python rt2_demo.py
"""

import torch
import torch.nn as nn
import torch.nn.functional as F
import numpy as np
import math
from dataclasses import dataclass
from typing import List, Optional, Tuple

# ═══════════════════════════════════════════════════════════════
# CONSTANTS
# ═══════════════════════════════════════════════════════════════
VOCAB_SIZE    = 512    # toy (real: 250k+)
EMBED_DIM     = 128   # toy (real: 4096+)
N_HEADS       = 4
N_LAYERS      = 3
IMG_PATCHES   = 16    # toy (real ViT: 256+)
ACTION_BINS   = 256   # discretization bins per action dimension
ACTION_DIM    = 7     # [dx, dy, dz, drx, dry, drz, gripper]
ACTION_MIN    = -1.0
ACTION_MAX    =  1.0

# Toy vocabulary: regular words + action tokens
# In RT-2 PaLI-X: integers 0-255 each have a unique token
# Here we simply reserve the LAST 256 vocab IDs as action tokens
ACTION_TOKEN_OFFSET = VOCAB_SIZE - ACTION_BINS  # = 256
TERMINATE_TOKEN     = 510   # special "stop" token
EOS_TOKEN           = 511   # end of sequence
BOS_TOKEN           = 1
PAD_TOKEN           = 0

# Words used in our toy vocab for prompts
WORD_VOCAB = {'<pad>':0, '<bos>':1, 'q':2, 'what':3, 'action':4,
             'should':5, 'robot':6, 'take':7, 'to':8, 'pick':9,
             'up':10, 'apple':11, 'a':12}

def tokenize(text):
    return [WORD_VOCAB.get(w.lower(), 0) for w in text.split()]


# ═══════════════════════════════════════════════════════════════
# ACTION ENCODING / DECODING
# ═══════════════════════════════════════════════════════════════

def continuous_to_bins(action: np.ndarray) -> List[int]:
    """Discretize a continuous action vector to bin indices [0, 255]."""
    action   = np.clip(action, ACTION_MIN, ACTION_MAX)
    bins     = (action - ACTION_MIN) / (ACTION_MAX - ACTION_MIN) * (ACTION_BINS - 1)
    return bins.astype(int).tolist()

def bins_to_continuous(bins: List[int]) -> np.ndarray:
    """Reverse: bin indices → continuous action values."""
    b = np.array(bins, dtype=np.float32)
    return b / (ACTION_BINS - 1) * (ACTION_MAX - ACTION_MIN) + ACTION_MIN

def action_to_token_ids(action: np.ndarray, terminate: bool = False) -> List[int]:
    """
    Convert a 7-dim action vector to a list of vocabulary token IDs.
    Token IDs for actions are offset from ACTION_TOKEN_OFFSET.
    Terminate flag uses TERMINATE_TOKEN.
    """
    bins = continuous_to_bins(action)
    # Each bin index b maps to vocabulary token (ACTION_TOKEN_OFFSET + b)
    action_tokens = [ACTION_TOKEN_OFFSET + b for b in bins]
    terminate_tok = [TERMINATE_TOKEN] if terminate else [ACTION_TOKEN_OFFSET + 1]
    return terminate_tok + action_tokens   # 8 tokens total

def token_ids_to_action(token_ids: List[int]) -> Tuple[bool, np.ndarray]:
    """Decode a sequence of token IDs back to a robot action."""
    terminate = (token_ids[0] == TERMINATE_TOKEN)
    bins      = [t - ACTION_TOKEN_OFFSET for t in token_ids[1:]]
    action    = bins_to_continuous(bins)
    return terminate, action

# ── Quick test ────────────────────────────────────────────────────────
test_action = np.array([0.05, -0.12, 0.03, 0.05, 0.02, 0.08, 0.3])
tids = action_to_token_ids(test_action, terminate=False)
print(f"Action token IDs: {tids}")   # [257, 390, 369, 386, 390, 387, 394, 420]
t, a = token_ids_to_action(tids)
print(f"Recovered: terminate={t}, action={np.round(a, 3)}")


# ═══════════════════════════════════════════════════════════════
# THE MODEL: Vision-Language-Action (VLA)
# ═══════════════════════════════════════════════════════════════

class PositionalEncoding(nn.Module):
    def __init__(self, max_len=512):
        super().__init__()
        pe  = torch.zeros(max_len, EMBED_DIM)
        pos = torch.arange(max_len).unsqueeze(1).float()
        div = torch.exp(torch.arange(0, EMBED_DIM, 2).float() *
                        (-math.log(10000.0) / EMBED_DIM))
        pe[:, 0::2] = torch.sin(pos * div)
        pe[:, 1::2] = torch.cos(pos * div)
        self.register_buffer('pe', pe.unsqueeze(0))

    def forward(self, x):
        return x + self.pe[:, :x.shape[1]]


class TinyVLAModel(nn.Module):
    """
    RT-2 in miniature.

    Simulates the core VLA architecture:
    - Image encoder (fake ViT patch projection)
    - Shared transformer decoder (handles BOTH text and action outputs)
    - Output constraint: at inference, only sample valid action tokens
    """
    def __init__(self):
        super().__init__()

        # Shared embedding table for ALL tokens (text + action)
        self.embedder  = nn.Embedding(VOCAB_SIZE, EMBED_DIM, padding_idx=0)
        self.pos_enc   = PositionalEncoding()

        # Image encoder: in reality a ViT-22B / ViT-4B
        # Here: linear projection from random 32-dim patch features to EMBED_DIM
        self.img_enc   = nn.Sequential(
            nn.Linear(32, EMBED_DIM),
            nn.GELU(),
            nn.Linear(EMBED_DIM, EMBED_DIM),
        )

        # The main transformer: decoder-only (causal attention)
        encoder_layer = nn.TransformerEncoderLayer(
            d_model=EMBED_DIM, nhead=N_HEADS,
            dim_feedforward=EMBED_DIM*4, dropout=0.0,
            batch_first=True, norm_first=True
        )
        self.transformer = nn.TransformerEncoder(encoder_layer, num_layers=N_LAYERS)

        # Output head: project to vocab logits
        self.out_head = nn.Linear(EMBED_DIM, VOCAB_SIZE, bias=False)
        self.out_head.weight = self.embedder.weight  # weight tying

    def _build_prefix(self, img_patches, text_token_ids):
        """Build the multimodal prefix: [image tokens | text tokens]."""
        img_emb  = self.img_enc(img_patches)          # (B, IMG_PATCHES, D)
        text_emb = self.embedder(text_token_ids)       # (B, T_text, D)
        prefix   = torch.cat([img_emb, text_emb], 1)  # (B, IMG_PATCHES+T, D)
        return self.pos_enc(prefix)

    def forward(self, img_patches, text_ids, target_ids):
        """
        Forward pass for training.
        Both VQA targets (text token IDs) and action targets
        (action token IDs) pass through identically.
        """
        B, T_tgt = target_ids.shape

        prefix     = self._build_prefix(img_patches, text_ids)
        target_emb = self.pos_enc(self.embedder(target_ids))

        full_seq = torch.cat([prefix, target_emb], 1)
        T_full   = full_seq.shape[1]

        # Causal mask: no peeking at future tokens
        causal_mask = torch.triu(
            torch.ones(T_full, T_full, device=full_seq.device), diagonal=1
        ).bool()

        out        = self.transformer(full_seq, mask=causal_mask, is_causal=True)
        T_prefix   = prefix.shape[1]
        logits     = self.out_head(out[:, T_prefix:, :])  # (B, T_tgt, VOCAB)
        return logits

    @torch.no_grad()
    def generate_action(self, img_patches, text_ids, constrain_to_actions=True):
        """
        Generate one full robot action (8 tokens) given image + instruction.

        constrain_to_actions: if True, only sample from valid action token IDs.
          This is the "output constraint" described in the paper.
          Prevents the model from outputting "apple" when it should output "128".
        """
        self.eval()
        B      = img_patches.shape[0]
        prefix = self._build_prefix(img_patches, text_ids)

        # Action token IDs: TERMINATE_TOKEN + ACTION_TOKEN_OFFSET to ACTION_TOKEN_OFFSET+255
        valid_action_ids = torch.tensor(
            [TERMINATE_TOKEN] + list(range(ACTION_TOKEN_OFFSET, ACTION_TOKEN_OFFSET + ACTION_BINS))
        )

        generated = []
        # Generate exactly ACTION_DIM + 1 tokens (terminate + 7 dims)
        current_seq = prefix

        for step in range(ACTION_DIM + 1):
            T_cur = current_seq.shape[1]
            causal_mask = torch.triu(
                torch.ones(T_cur, T_cur, device=current_seq.device), diagonal=1
            ).bool()

            out    = self.transformer(current_seq, mask=causal_mask, is_causal=True)
            logits = self.out_head(out[:, -1, :])  # (B, VOCAB)

            if constrain_to_actions:
                # ★ OUTPUT CONSTRAINT: zero out all non-action logits ★
                # This guarantees the model only picks valid action token IDs
                mask = torch.full_like(logits, float('-inf'))
                mask[:, valid_action_ids] = logits[:, valid_action_ids]
                logits = mask

            probs     = torch.softmax(logits, dim=-1)
            next_tok  = torch.multinomial(probs, 1)  # (B, 1)
            generated.append(next_tok)

            # Append new token to sequence
            next_emb    = self.pos_enc(self.embedder(next_tok))  # (B, 1, D)
            current_seq = torch.cat([current_seq, next_emb], dim=1)

        # Stack all generated token IDs: (B, ACTION_DIM+1)
        return torch.cat(generated, dim=1)

def make_training_batch(batch_size=8, robot_ratio=0.5):
    """
    Create a mixed training batch with robot and VQA examples.
    robot_ratio: fraction of batch that is robot data.
    """
    n_robot = int(batch_size * robot_ratio)
    n_vqa   = batch_size - n_robot

    examples = []

    # Robot examples: action tokens as target
    for _ in range(n_robot):
        img     = torch.randn(IMG_PATCHES, 32)            # fake patches
        text    = torch.tensor([2, 3, 4, 5, 6, 7, 8, 9]) # "q what action should robot take to pick"
        raw_act = np.random.uniform(-1, 1, ACTION_DIM)    # random action
        tgt_ids = torch.tensor(action_to_token_ids(raw_act, terminate=False))
        examples.append((img, text, tgt_ids, 'robot'))

    # VQA examples: text tokens as target
    for _ in range(n_vqa):
        img     = torch.randn(IMG_PATCHES, 32)
        text    = torch.tensor([2, 3, 4, 12])         # "q what a ..."
        tgt_ids = torch.randint(2, ACTION_TOKEN_OFFSET, (6,))  # random words (not action tokens)
        examples.append((img, text, tgt_ids, 'vqa'))

    return examples

def pad_batch(examples):
    """Collate examples to same length for batch processing."""
    max_text = max(e[1].shape[0] for e in examples)
    max_tgt  = max(e[2].shape[0] for e in examples)

    imgs, texts, tgts = [], [], []
    for img, text, tgt, _ in examples:
        imgs.append(img)
        pad_t = F.pad(text, (0, max_text - text.shape[0]), value=PAD_TOKEN)
        pad_g = F.pad(tgt,  (0, max_tgt  - tgt.shape[0]),  value=PAD_TOKEN)
        texts.append(pad_t)
        tgts.append(pad_g)

    return (torch.stack(imgs), torch.stack(texts), torch.stack(tgts))

class ToyRobotEnv:
    """
    Simulates a simple robot environment.
    State: end-effector position [x, y, z]
    Goal: reach [0.5, 0.5, 0.5]
    """
    def __init__(self):
        self.pos     = np.array([0.0, 0.0, 0.0])
        self.goal    = np.array([0.5, 0.5, 0.5])
        self.success = False
        self.steps   = 0

    def get_obs(self):
        """Return a fake camera image (noise + position encoded)."""
        # In reality: RGB image from robot camera
        fake_img = torch.randn(1, IMG_PATCHES, 32)
        fake_img[0, 0, :3] = torch.tensor(self.pos)  # encode pos into first patch
        return fake_img

    def step(self, delta_pos):
        """Apply a position delta and check if goal is reached."""
        self.pos   += delta_pos[:3]  # only use first 3 dims (x, y, z)
        self.steps += 1
        dist = np.linalg.norm(self.pos - self.goal)
        self.success = dist < 0.1
        return self.success, dist

⚔️ RT-2 vs PaLM-E: What's the Difference?

Both papers came from Google in 2023, both use giant VLMs for robotics, but they make fundamentally different choices about how to connect language to motion. Understanding the difference is crucial.

Who wins?

RT-2 is more elegant and more end-to-end, but it's also limited: the VLM can only plan within the physical skills it was trained on. PaLM-E is more modular — you can swap in a better low-level policy without retraining the whole VLM. Both approaches coexist in the literature, and modern systems (like RT-X) draw on both ideas.

🎯 Key Takeaways

1

Actions Are Just Another Language

By discretizing continuous motor commands into bins and expressing them as text tokens, actions and words share the same model, the same loss function, and the same weights. This is the paper's single most important idea.

2

Co-Fine-Tuning is Non-Negotiable

Training only on robot data destroys the web knowledge. Training from scratch with robot data fails entirely. You must mix robot + web data in every training batch — the paper shows this doubles generalization performance.

3

Emergence is Real

The model can "pick the object closest to another object" or "place the apple on the number 3" — tasks completely absent from robot training data. This reasoning comes entirely from web pretraining. Scale unlocks it: 55B > 5B significantly.

4

Chain-of-Thought Transfers

A technique invented for pure language models (think before you answer) directly works for robot control. The model can reason "I need a hammer → rock is the closest thing → pick rock" and then output the correct motor command.

5

Output Constraints at Inference

One practical engineering detail: when controlling a robot, mask out all non-action tokens so the model can only sample valid action bins. This is a simple but crucial detail for reliability.

6

New Skills Need New Motion Data

RT-2 inherits semantic knowledge but NOT new physical motions. If the robot was only trained to pick and place, it can't suddenly learn to fold laundry from web data alone. Web knowledge helps WHERE and WHAT to pick, not HOW to move.

The 30-Second Summary

RT-2's insight is delightfully simple: robot actions and words are both just sequences of discrete symbols — so why not use the same model for both? Discretize continuous motor commands into 256 bins, express them as text tokens, and fine-tune a 55-billion-parameter VLM to output them alongside regular language. The result: a robot that can figure out to pick up a rock when asked to hammer a nail, because it read about rocks and hammers on the internet — and now that knowledge directly moves its arm.
