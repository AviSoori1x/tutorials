🤔 What Problem Does RT-2 Solve?

The trajectory of modern AI has been spectacular: Vision-Language Models (VLMs) trained on internet-scale data can describe scenes in French, identify obscure objects, answer complex visual questions, and reason about cause and effect. They have, in a meaningful sense, read the entire web. They know that rocks are hard and dense — useful as improvised hammers. They know energy drinks are for tired people. They know strawberries belong with fruit, not vegetables.

Robots, meanwhile, are extraordinarily literal. A robot arm doesn't care about "knowing" things. At each timestep it needs a concrete 7-dimensional vector: how many millimeters to translate the end-effector in X, Y, Z; how many degrees to rotate around each axis; how open the gripper should be; and whether to terminate. These seem completely incompatible with the text-token outputs of a VLM.

The Core Disconnect

Every prior approach used a separate low-level controller to bridge this gap. The VLM would produce a text command like "move towards the apple" and a separate neural policy — trained only on robot trajectories — would execute it. The problem: the VLM's rich world knowledge never directly influences the motor commands. The web knowledge sits upstairs; the motors are in the basement; and there's no staircase.

What was tried before RT-2?

VLM as High-Level Planner (SayCan, PaLM-E)

The VLM outputs step-by-step text instructions. A completely separate low-level policy (RT-1, SayCan's skills) then executes each step. Two separate models, two separate training regimes. Web knowledge can determine what to do but never how to move.

Pretrained Vision Encoder Only (VC-1, R3M)

Borrow the VLM's visual encoder (e.g. CLIP-style) to compute image embeddings, then feed those into a small policy network. Loses most of the VLM's language reasoning. You get a better camera; you don't get the web's knowledge about physics, objects, or human intent.

VLM as Semantic Map (MOO)

Use a VLM to generate an additional "semantic image channel" (highlighting relevant objects), then pass this augmented image into an RT-1 backbone. Better than a plain vision encoder, but still a two-system architecture with a knowledge bottleneck between them.

Train from Scratch on Robot Data

Design a new architecture (like Gato or RT-1) and train it purely on robot demonstrations. This yields no generalization beyond the training distribution. You can't pick an object the model hasn't seen; "rock as hammer" is unthinkable.

RT-2's Radical Simplification

What if actions were text? Not an abstract representation of actions, not a neural code for actions — literally text tokens representing discretized bin numbers. Then a VLM that already knows how to generate text can output motor commands using the exact same mechanism, parameters, and training procedure it uses for language. No new modules. No translation layer. One model, one loss, everything shared.

Why shared weights matter so much

When the model weights are unified across language, vision, and actions, the gradient signal from robot trajectory loss flows back through the same parameters that encode world knowledge. Web knowledge doesn't have to be "distilled" into a separate system — it is directly available at every layer when producing motor tokens. This is the fundamental shift that enables emergence.

The formal objective

RT-2 is trained with the standard autoregressive language modeling objective, but the "vocabulary" now includes both natural language tokens and action bin tokens:

Autoregressive factorization over the action string: p(a₁, a₂, ..., aₙ | image, instruction) = ∏ᵢ p(aᵢ | a₁...aᵢ₋₁, image, instruction) where each aᵢ is an action token drawn from the combined vocabulary: V = V_language ∪ V_actions For PaLI-X: V_actions = { "1", "2", ..., "256" } (existing integer tokens) For PaLM-E: V_actions = 256 least-frequent tokens (overwritten with action meaning) Training loss (same as standard language modeling): L = -Σᵢ log p(aᵢ | a₁...aᵢ₋₁, image, instruction) No new parameters. No new loss terms. Actions are just tokens.

🎮 Concept 1: Robot Actions as Text Tokens

The action space

RT-2 uses the same action space as RT-1 (Brohan et al. 2022), applied to a 7-DoF mobile manipulator (Everyday Robots arm). At each timestep the robot outputs 8 numbers:

Each action is a tuple of 7 discretized dimensions + a termination flag:

1terminate?
→
128Δpos_x
91Δpos_y
241Δpos_z
5Δrot_x
101Δrot_y
127Δrot_z
217gripper

Target string sent to the VLM: "1 128 91 241 5 101 127 217"

Discretization: 256 uniform bins

Each continuous dimension (Δx, Δy, Δz, Δroll, Δpitch, Δyaw, gripper) is discretized into 256 uniformly-spaced bins. The discretization follows RT-1's scheme:

PYTHON — Exact discretization logic (from RT-1/RT-2)
import numpy as np

class ActionTokenizer:
    """
    Converts continuous 7-DoF end-effector actions to/from
    space-separated integer strings for VLM fine-tuning.

    Action dimensions:
      [terminate, Δpos_x, Δpos_y, Δpos_z, Δrot_x, Δrot_y, Δrot_z, gripper]

    Each continuous dim → 256 uniform bins → integer string token.
    """

    def __init__(self,
                 action_min: np.ndarray,  # per-dim min of training set
                 action_max: np.ndarray,  # per-dim max of training set
                 num_bins: int = 256):
        self.num_bins = num_bins
        self.action_min = action_min
        self.action_max = action_max
        self.bin_edges = np.linspace(action_min, action_max,
                                       num_bins + 1)  # shape [257, D]

    def encode(self, action: np.ndarray) -> str:
        """
        action: float array [D]  (continuous motor command)
        Returns: "1 128 91 241 5 101 127 217" style string
        """
        # clip to valid range
        action = np.clip(action, self.action_min, self.action_max)

        # find which bin each value falls into (0-indexed)
        bin_indices = np.digitize(action, self.bin_edges[1:-1], right=False)
        # shift to 1-indexed so "0" can be reserved / avoided
        bin_indices = bin_indices + 1

        return " ".join(str(b) for b in bin_indices)

    def decode(self, token_string: str) -> np.ndarray:
        """
        token_string: "1 128 91 241 5 101 127 217"
        Returns: float array [D] of continuous motor commands
        """
        tokens = [int(t) for t in token_string.split()]
        bin_indices = np.array(tokens) - 1  # back to 0-indexed

        # midpoint of each bin as the decoded value
        midpoints = (self.bin_edges[bin_indices] +
                     self.bin_edges[bin_indices + 1]) / 2.0
        return midpoints

    def action_to_prompt(self, task: str, action: np.ndarray) -> dict:
        """Build VQA-style training example."""
        return {
            "prompt": f"Q: what action should the robot take to {task}? A:",
            "target": self.encode(action),
        }


# ── Example usage ──────────────────────────────────────────────────────
action_min = np.array([0, -0.05, -0.05, -0.05, -15, -15, -15, 0.0])
action_max = np.array([1,  0.05,  0.05,  0.05,  15,  15,  15, 1.0])
tokenizer = ActionTokenizer(action_min, action_max)

# A real motor command: no-op (terminate), small x-move, slight rotation
action = np.array([1.0, 0.02, -0.01, 0.0, 5.0, 0.0, -3.0, 0.5])
encoded = tokenizer.encode(action)
print(encoded)  # → "1 179 114 128 196 128 83 128"

# Decode back to continuous
decoded = tokenizer.decode(encoded)
print(np.allclose(action, decoded, atol=0.001))  # → True (within bin width)

Token vocabulary assignment: PaLI-X vs PaLM-E

The two VLM backbones handle the action vocabulary differently, which is a subtle but important implementation detail:

PaLI-X (5B / 55B)

The PaLI-X tokenizer assigns unique tokens to integers up to 1000. So bins 1–256 directly map to the existing tokens "1", "2", …, "256". No vocabulary modification needed. The model already knows these tokens from language pretraining, which is why it can absorb action semantics with relatively few gradient steps.

PaLM-E (12B)

PaLM-E's tokenizer doesn't give single tokens to all integers. Instead, the 256 least-frequently used tokens in the vocabulary are overwritten to become action tokens. This is a form of symbol tuning (Wei et al. 2023) — reusing existing vocabulary slots for new semantics. The model re-learns these tokens' meanings during fine-tuning.

Why this design choice matters

Using PaLI-X's integer tokens means the model arrives at fine-tuning already having some "sense" of ordering and magnitude for these tokens — it has seen "128" is roughly the middle of 1–256, "1" is small, "256" is large. This prior may accelerate learning of the bin ordering semantics. PaLM-E's symbol tuning starts cold but benefits from overwriting rare tokens (which contribute little to language performance).

🏗️ Architecture: Two VLM Backbones

RT-2 is not a single model — it is a fine-tuning recipe applied to two different backbone VLMs. Both share the same action tokenization approach but differ substantially in their internal architectures.

RT-2-PaLI-X (5B and 55B)

PaLI-X (Chen et al. 2023) is a multilingual vision-language model built from two separately pretrained components joined by cross-attention:

The VLM IS the policy — no new parameters

Crucially, RT-2 adds zero new parameters to the VLM backbone. Unlike approaches that bolt on an action head or add cross-modal adapters, RT-2 simply fine-tunes the existing VLM output layer to generate action tokens instead of (or alongside) text tokens. This means 100% of the model's capacity — including the vision encoder, the language model, all attention heads — is shared and jointly optimized for both tasks.

🔀 Concept 2: Co-Fine-Tuning (The Secret Sauce)

The most important training decision in RT-2 isn't the architecture — it's how the data is mixed. This is the difference between a policy that generalizes to the web and one that forgets it.

Three training strategies, three very different outcomes

A

Train from scratch on robot data only

Initialize randomly. Train only on robot trajectory data (~130k demonstrations). Result: disaster. Even a 5B model trained this way achieves ~12% on generalization tasks. There are simply not enough robot demonstrations (relative to model capacity) to learn any useful representations from scratch. The model memorizes specific trajectory patterns but generalizes to nothing.

B

Fine-tune pretrained VLM on robot data only

Start from pretrained VLM weights (billions of params of internet knowledge). Fine-tune exclusively on robot data. Result: mediocre. ~38% on generalization tasks. Catastrophic forgetting partially destroys web knowledge because the optimizer keeps pushing the weights toward the robot distribution with no counterbalancing signal from language data. The model partially retains some priors but loses most of the semantic reasoning that made the VLM powerful.

C

Co-fine-tune: robot data + original VQA web data simultaneously (RT-2)

Mix robot trajectories with the original VQA/captioning web data in every training batch. The model sees both continuously. Catastrophic forgetting cannot happen because the gradient signal from language tasks constantly reinforces the web knowledge. Result: ~62% on generalization tasks — 5× better than training from scratch.

(2) Scale helps significantly — but co-tuning matters more.

Data mixture and sampling weights

The RT-1 training dataset (reused for RT-2) contains ~130k robot demonstrations collected with 13 robots over 17 months in office kitchen environments. Tasks include picking, placing, opening/closing drawers, knocking objects over, and more — 700+ instructions total.

The robot data is dramatically smaller than the VQA web data (tens of millions of examples). Without explicit upsampling, a robot example would almost never appear in a batch:

PYTHON — Co-fine-tuning data pipeline with correct sampling weights
import torch
from torch.utils.data import Dataset, ConcatDataset, WeightedRandomSampler, DataLoader
import numpy as np

class RT1RobotDataset(Dataset):
    """
    RT-1 robot demonstration dataset.
    ~130k episodes, 700+ tasks, 13 robots, 17 months collection.
    Each item: (image, task_instruction, action_string).
    """
    def __init__(self, paths, tokenizer: ActionTokenizer):
        self.episodes = self.load_episodes(paths)   # RLDS format
        self.tokenizer = tokenizer

    def __getitem__(self, idx):
        ep = self.episodes[idx]
        # Each episode: sample a single (image, timestep, action) pair
        t = np.random.randint(0, len(ep['actions']))
        image   = ep['images'][t]         # (3, 224, 224) tensor
        task    = ep['instruction']       # e.g. "pick up the 7up can"
        action  = ep['actions'][t]        # (8,) float array
        encoded = self.tokenizer.encode(action)

        return {
            'image':  image,
            'prompt': f"Q: what action should the robot take to {task}? A:",
            'target': encoded,   # e.g. "1 128 91 241 5 101 127 217"
            'source': 'robot'
        }


class WebVQADataset(Dataset):
    """
    Internet-scale VQA + captioning data.
    Identical format to RobotDataset — outputs are just natural language answers
    instead of action strings. This is what allows unified training.
    """
    def __getitem__(self, idx):
        item = self.data[idx]
        return {
            'image':  item['image'],
            'prompt': item['question'],
            'target': item['answer'],   # e.g. "a red apple on a white plate"
            'source': 'vqa'
        }


# ── Build co-fine-tuning dataloader ────────────────────────────────────

robot_dataset = RT1RobotDataset(paths=..., tokenizer=tokenizer)   # ~130k
web_dataset   = WebVQADataset(...)                                   # ~10M+

combined = ConcatDataset([robot_dataset, web_dataset])

# ── CRITICAL: upsample robot data so it fills ~50% of every batch ──────
# Without this, robot data appears in <1% of batches (10k / 10M).
# Paper targets approximately equal representation.

N_robot = len(robot_dataset)
N_web   = len(web_dataset)

# Each robot sample gets weight proportional to N_web / N_robot
robot_weight = 1.0                    # base weight
web_weight   = N_robot / N_web         # ~0.013 → robot seen as frequently as web

sample_weights = (
    [robot_weight] * N_robot +
    [web_weight]   * N_web
)

sampler = WeightedRandomSampler(
    weights=sample_weights,
    num_samples=len(combined),
    replacement=True
)

loader = DataLoader(
    combined,
    batch_size=256,      # large batch for TPU efficiency
    sampler=sampler,
    num_workers=16,
    pin_memory=True
)

# ── Training loop ─────────────────────────────────────────────────────
# Same cross-entropy loss for BOTH robot and VQA batches.
# No separate loss terms, no task identifiers, no routing.
# The model sees actions and text as the same thing.

for batch in loader:
    images  = batch['image'].to(device)    # (B, 3, 224, 224)
    prompts = batch['prompt']              # list of strings
    targets = batch['target']              # list of strings (actions or text)

    # Tokenize prompts + targets, forward pass, compute token-level cross-entropy
    loss = model.forward_loss(images, prompts, targets)
    loss.backward()
    optimizer.step()
    optimizer.zero_grad()
Elegance of unified training

Notice that there is no task-specific loss, no router, no task embedding. The model simply sees a batch that sometimes has natural-language targets and sometimes has action-string targets. It learns to produce both under the same cross-entropy objective. The action tokens and language tokens live in the same vocabulary; the model implicitly learns to distinguish robot-action contexts (VQA format + robot camera image) from VQA contexts.

🧠 Concept 3: Chain-of-Thought Reasoning for Robots

One of RT-2's most striking contributions is that it can perform chain-of-thought reasoning — a technique developed entirely for language models — and apply it directly to physical control. This works because the VLM backbone can generate free-form text before outputting action tokens.

How it works: augmented data with plan tokens

The CoT variant of RT-2 is fine-tuned for a few hundred additional gradient steps on data augmented with a "Plan" field. Instead of the format:

Standard RT-2 format: ───────────────────────────────────────────────────────────────── Q: what action should the robot take to [instruction]? A: [action_tokens] Chain-of-Thought RT-2 format: ───────────────────────────────────────────────────────────────── Q: what action should the robot take to [instruction]? Plan: [natural language reasoning step(s)] Action: [action_tokens] Example from the paper: ───────────────────────────────────────────────────────────────── Instruction: "What can I use to hammer a nail?" Plan: "The best object to use as an improvised hammer is a rock because it is hard and dense. I see a rock on the table." Action: "1 089 143 227 12 101 118" ← pick up the rock Another example: ───────────────────────────────────────────────────────────────── Instruction: "I'm tired. Get me a drink." Plan: "Energy drinks help with fatigue. I see a Monster can on the counter. I should pick that up." Action: "1 114 098 201 08 127 133" ← pick up the energy drink

Critically, this involves visually-grounded planning: the reasoning step refers to objects the model actually sees in the camera image ("I see a rock on the table"), not just abstract language reasoning. This is something SayCan and other purely language-based planners cannot do.

Why this is remarkable

Chain-of-thought prompting was invented to improve LLM performance on math and reasoning tasks. RT-2 inherits this technique for free — because the VLM backbone already supports it, and because the shared weights mean that language reasoning and action generation are deeply coupled. You can literally prompt RT-2 to "think before you act" by prepending "Plan:" to the output format, and the quality of motor commands improves accordingly.

What kinds of reasoning does CoT RT-2 unlock?

Object affordance reasoning

"Which object to use as a hammer?" → identifies rock's hardness from web knowledge → grips rock. The model connects physical properties learned from text to visual objects in the scene.

Human intent inference

"I'm tired" → infers this implies caffeine need → picks energy drink. Social/contextual inference from web training directly influences motor output.

Multi-step skill sequencing

Long-horizon commands like "tidy up the table" require the model to enumerate sub-steps in natural language (Plan: step 1, step 2…) before producing action tokens for each step.

⚡ Concept 4: Real-Time Inference Engineering

Deploying a 55B-parameter model for real-time robot control is a non-trivial infrastructure challenge. RT-2 required genuinely new solutions that are worth understanding in detail.

The challenge: latency vs model size

Model sizes and frequencies

RT-2-PaLI-X 55B: 1–3 Hz (limited by network RTT + TPU inference latency)

RT-2-PaLI-X 5B: ~5 Hz (suitable for slower manipulation tasks)

RT-1 (35M parameters): ~3 Hz on-robot GPU

Note: RT-2 at 55B is >1,000× larger than RT-1, yet achieves comparable inference frequency — owing entirely to TPU cloud offload.

Cloud TPU deployment

RT-2 runs on multi-TPU cloud pods (GCP), with the robot querying the service over a local network connection. This allows one cloud service to serve multiple robots simultaneously, amortizing the hardware cost across a fleet. The tradeoff: the model is not local to the robot, so network reliability is critical.

Output constraint: ensuring valid actions during decoding

When the VLM is deployed in robot-control mode, it must output valid action tokens — not words like "the" or "apple". This is enforced at the decoding level:

PYTHON — Constrained decoding for robot action mode
import torch
import torch.nn.functional as F

class RT2InferenceEngine:
    def __init__(self, model, tokenizer, action_token_ids: list[int]):
        """
        model:            the fine-tuned VLA (PaLI-X or PaLM-E backbone)
        tokenizer:        the VLM tokenizer
        action_token_ids: list of 256 token IDs reserved for actions
                          (e.g. [tokenizer.encode("1"), ..., tokenizer.encode("256")]
                           for PaLI-X, or the 256 LFU token IDs for PaLM-E)
        """
        self.model = model
        self.tokenizer = tokenizer
        self.action_mask = self.build_action_mask(action_token_ids)

    def build_action_mask(self, action_token_ids):
        """Binary mask: 1 for action tokens, 0 for everything else."""
        mask = torch.full((self.tokenizer.vocab_size,), -1e9)  # block all
        mask[action_token_ids] = 0.0                            # allow action IDs
        return mask  # added to logits before softmax

    def predict_action(self, image: torch.Tensor, task: str,
                        mode: str = "robot") -> np.ndarray:
        """
        image: (3, 224, 224) camera observation
        task:  natural language instruction
        mode:  "robot" → constrain to action tokens
               "vqa"   → unconstrained (full vocabulary)
        Returns: decoded continuous motor command (8,)
        """
        prompt = f"Q: what action should the robot take to {task}? A:"
        input_ids = self.tokenizer.encode(prompt, return_tensors="pt")

        generated_tokens = []
        num_action_dims = 8   # terminate + 6-DoF + gripper

        with torch.no_grad():
            for step in range(num_action_dims):
                logits = self.model.forward(
                    images=image.unsqueeze(0),
                    input_ids=input_ids
                ).logits[:, -1, :]  # (1, vocab_size) — last token position

                if mode == "robot":
                                        logits = logits + self.action_mask.to(logits.device)
                # ↑ This is the key: adds -1e9 to all non-action token logits,
                #   making them effectively impossible to sample.
                #   Equivalent to masking out the full vocabulary except 256 bins.

                next_token = logits.argmax(dim=-1)  # greedy decoding
                generated_tokens.append(next_token.item())

                # append generated token as new input for next step
                input_ids = torch.cat([input_ids,
                                        next_token.unsqueeze(0).unsqueeze(0)], dim=1)

        # decode token IDs → bin indices → continuous actions
        token_string = " ".join(self.tokenizer.decode([t]) for t in generated_tokens)
        return action_tokenizer.decode(token_string)   # → (8,) float array


# ── Closed-loop control at 1–3 Hz ─────────────────────────────────────
engine = RT2InferenceEngine(model, tokenizer, action_token_ids)

while not task_done:
    image   = robot.get_camera_image()              # latest frame
    action  = engine.predict_action(image, task)     # ~300–1000ms round-trip

    terminate = action[0] > 0.5
    if terminate:
        break

    robot.execute_action(
        delta_pos = action[1:4],   # Δx, Δy, Δz in meters
        delta_rot = action[4:7],   # Δroll, Δpitch, Δyaw in degrees
        gripper   = action[7]       # 0.0 = closed, 1.0 = open
    )
Output constraint is not optional

Without the action mask, the model sometimes outputs natural-language words instead of action numbers, especially when the VQA distribution bleeds into the robot-control mode. A VLM that outputs "pick up" instead of "1 128 91..." crashes the robot. The constraint is a simple logit-masking operation — no retraining needed — but it's a required component of any RT-2 deployment.

Below is a complete, reference-quality implementation of the core RT-2 training and inference components, drawing from the paper's appendix details and the OpenVLA open-source implementation (which replicates RT-2's approach with public weights).

PYTHON — Complete RT-2 / VLA training system
"""
RT-2 / Vision-Language-Action Model — Full Reference Implementation
Based on: Brohan et al. 2023 (arXiv:2307.15818)
Informed by: OpenVLA (Kim et al. 2024, github.com/openvla/openvla)

Covers:
  1. Action discretization + tokenization
  2. Dataset construction (robot + web)
  3. Co-fine-tuning loop
  4. Constrained inference for closed-loop robot control
"""

import torch
import torch.nn as nn
import numpy as np
from torch.utils.data import Dataset, ConcatDataset, WeightedRandomSampler, DataLoader
from transformers import AutoTokenizer, AutoModelForVision2Seq
from dataclasses import dataclass
from typing import Optional
import logging

# ═══════════════════════════════════════════════════════════════════════
# 1. ACTION SPACE CONFIGURATION
# ═══════════════════════════════════════════════════════════════════════

@dataclass
class ActionConfig:
    """
    7-DoF end-effector delta control, following RT-1 convention.
    Dimensions: [terminate, Δx, Δy, Δz, Δroll, Δpitch, Δyaw, gripper]
    """
    # Physical limits (meters for translation, degrees for rotation)
    pos_delta_limit: float  = 0.05     # ±50mm per timestep
    rot_delta_limit: float  = 15.0     # ±15° per timestep
    num_bins:        int    = 256
    num_dims:        int    = 8         # terminate + 6-DoF + gripper

    @property
    def action_min(self) -> np.ndarray:
        return np.array([0,
                          -self.pos_delta_limit, -self.pos_delta_limit, -self.pos_delta_limit,
                          -self.rot_delta_limit, -self.rot_delta_limit, -self.rot_delta_limit,
                          0.0])

    @property
    def action_max(self) -> np.ndarray:
        return np.array([1,
                          self.pos_delta_limit, self.pos_delta_limit, self.pos_delta_limit,
                          self.rot_delta_limit, self.rot_delta_limit, self.rot_delta_limit,
                          1.0])


class ActionTokenizer:
    """Converts between continuous motor commands and action token strings."""

    def __init__(self, cfg: ActionConfig, vlm_tokenizer):
        self.cfg = cfg
        self.vlm_tokenizer = vlm_tokenizer
        # Bin edges: shape [num_bins+1, num_dims]
        self.bin_edges = np.stack([
            np.linspace(lo, hi, cfg.num_bins + 1)
            for lo, hi in zip(cfg.action_min, cfg.action_max)
        ], axis=1)   # [257, 8]
        # Token IDs that represent action bins in the VLM vocabulary
        self.action_token_ids = self._resolve_action_token_ids()

    def _resolve_action_token_ids(self):
        """For PaLI-X style: integers 1-256 have unique token IDs."""
        ids = []
        for i in range(1, self.cfg.num_bins + 1):
            encoded = self.vlm_tokenizer.encode(str(i), add_special_tokens=False)
            assert len(encoded) == 1, f"Token '{i}' maps to multiple subwords!"
            ids.append(encoded[0])
        return ids  # list of 256 token IDs

    def encode(self, action: np.ndarray) -> str:
        action = np.clip(action, self.cfg.action_min, self.cfg.action_max)
        # Digitize each dimension independently
        bin_idx = np.array([
            np.digitize(action[d], self.bin_edges[:, d][1:-1], right=False) + 1
            for d in range(self.cfg.num_dims)
        ])
        return " ".join(map(str, bin_idx))

    def decode(self, token_string: str) -> np.ndarray:
        bins = np.array([int(t) for t in token_string.split()])
        idx  = bins - 1
        return (self.bin_edges[idx, range(self.cfg.num_dims)] +
                self.bin_edges[idx + 1, range(self.cfg.num_dims)]) / 2.0


# ═══════════════════════════════════════════════════════════════════════
# 2. VLA MODEL WRAPPER
# ═══════════════════════════════════════════════════════════════════════

class VisionLanguageActionModel(nn.Module):
    """
    Wraps any pretrained VLM (PaLI-X or PaLM-E style) for RT-2 training.
    No new parameters are added — actions are trained in the existing output head.
    """

    def __init__(self, vlm_name_or_path: str, action_tokenizer: ActionTokenizer):
        super().__init__()
        self.vlm = AutoModelForVision2Seq.from_pretrained(
            vlm_name_or_path,
            torch_dtype=torch.bfloat16,   # bfloat16 for TPU efficiency
            device_map="auto"
        )
        self.action_tokenizer = action_tokenizer
        # Build the logit mask for inference-time output constraint
        vocab_size = self.vlm.config.vocab_size
        self.action_logit_bias = torch.full((vocab_size,), -1e9)
        self.action_logit_bias[action_tokenizer.action_token_ids] = 0.0

    def forward(self, pixel_values, input_ids, attention_mask,
                  labels=None) -> torch.Tensor:
        """
        Standard VLM forward pass.
        Labels contain token IDs of action strings — cross-entropy loss computed
        only on non-padding label positions (same as standard language modeling).
        """
        output = self.vlm(
            pixel_values   = pixel_values,
            input_ids      = input_ids,
            attention_mask = attention_mask,
            labels         = labels         # None → just logits; not None → loss
        )
        return output

    @torch.no_grad()
    def generate_action(self, pixel_values, input_ids,
                          attention_mask, use_constraint: bool = True) -> np.ndarray:
        """Autoregressively generate the action token string."""
        num_dims = self.action_tokenizer.cfg.num_dims
        generated = []
        cur_ids    = input_ids
        cur_mask   = attention_mask

        for _ in range(num_dims):
            out = self.vlm(pixel_values=pixel_values,
                            input_ids=cur_ids, attention_mask=cur_mask)
            logits = out.logits[:, -1, :]   # (B, vocab)

            if use_constraint:
                logits = logits + self.action_logit_bias.to(logits.device)

            next_id = logits.argmax(-1, keepdim=True)  # greedy (B, 1)
            generated.append(next_id)
            cur_ids  = torch.cat([cur_ids, next_id], dim=1)
            cur_mask = torch.cat([cur_mask,
                                   torch.ones_like(next_id)], dim=1)

        token_ids   = torch.cat(generated, dim=1)[0].tolist()
        token_strs  = [self.action_tokenizer.vlm_tokenizer.decode([t]) for t in token_ids]
        action_str  = " ".join(token_strs)
        return self.action_tokenizer.decode(action_str)


# ═══════════════════════════════════════════════════════════════════════
# 3. CO-FINE-TUNING TRAINER
# ═══════════════════════════════════════════════════════════════════════

class CoFineTuningTrainer:
    """
    Implements RT-2's co-fine-tuning recipe:
    - Mix robot trajectory data with original VQA web data
    - Upsample robot data to achieve ~50% presence in each batch
    - Same cross-entropy loss for both data types
    - Use original VLM hyperparameters (LR schedule, regularization)
    """

    def __init__(self, model: VisionLanguageActionModel,
                  robot_dataset: Dataset, web_dataset: Dataset,
                  robot_fraction: float = 0.5,   # target fraction in each batch
                  batch_size: int = 256,
                  learning_rate: float = 2e-5,   # keep same as VLM pretraining
                  max_steps: int = 30_000):
        self.model = model
        self.max_steps = max_steps

        # Build combined dataset with proper sampling weights
        N_r, N_w  = len(robot_dataset), len(web_dataset)
        # weight_robot / weight_web = (robot_fraction/N_r) / ((1-robot_fraction)/N_w)
        w_robot = robot_fraction / N_r
        w_web   = (1 - robot_fraction) / N_w
        weights = [w_robot] * N_r + [w_web] * N_w

        sampler = WeightedRandomSampler(
            weights, num_samples=max(N_r, N_w) * 2, replacement=True)

        self.loader = DataLoader(
            ConcatDataset([robot_dataset, web_dataset]),
            batch_size=batch_size, sampler=sampler, num_workers=8,
            pin_memory=True
        )

        # Optimizer: AdamW with same LR as VLM pretraining (following paper)
        self.optimizer = torch.optim.AdamW(
            model.parameters(), lr=learning_rate,
            betas=(0.9, 0.95), weight_decay=0.1
        )
        # Cosine decay to 10% of peak LR over training
        self.scheduler = torch.optim.lr_scheduler.CosineAnnealingLR(
            self.optimizer, T_max=max_steps, eta_min=learning_rate * 0.1)

    def train(self):
        self.model.train()
        step = 0

        for batch in self.loader:
            if step >= self.max_steps: break

            # Unpack batch (same format for robot and VQA items)
            pixel_values   = batch['pixel_values'].to(device, dtype=torch.bfloat16)
            input_ids      = batch['input_ids'].to(device)
            attention_mask = batch['attention_mask'].to(device)
            labels         = batch['labels'].to(device)
            # labels: token IDs of target string, -100 for non-target positions
            # For robot items: labels = action token IDs
            # For VQA items:   labels = answer token IDs
            # Same loss function, no routing needed.

            output = self.model(pixel_values, input_ids, attention_mask, labels)
            loss   = output.loss

            self.optimizer.zero_grad()
            loss.backward()
            torch.nn.utils.clip_grad_norm_(self.model.parameters(), max_norm=1.0)
            self.optimizer.step()
            self.scheduler.step()

            if step % 100 == 0:
                sources = batch['source']
                n_robot = sum(1 for s in sources if s == 'robot')
                logging.info(f"Step {step} | loss: {loss.item():.4f} | robot%: {n_robot}/{len(sources)}")
            step += 1
PYTHON — Chain-of-Thought RT-2 inference
def predict_with_chain_of_thought(
        model: VisionLanguageActionModel,
        tokenizer, image: torch.Tensor, task: str) -> dict:
    """
    RT-2 CoT inference: model generates a natural-language Plan
    followed by action tokens.

    Output format trained on:
      Q: {task}?
      Plan: {reasoning steps...}
      Action: {action_tokens}
    """
    # ── Phase 1: Generate the plan (unconstrained, full vocabulary) ────────
    prompt = f"Q: {task}?\\nPlan:"
    plan_input = tokenizer(prompt, return_tensors="pt").to(device)

    plan_ids = model.vlm.generate(
        pixel_values=image.unsqueeze(0),
        **plan_input,
        max_new_tokens=50,          # allow up to 50 tokens of reasoning
        stop_token_ids=[# stop at "Action:" keyword
            tokenizer.encode("Action", add_special_tokens=False)[0]
        ],
        do_sample=False             # greedy for determinism
    )
    plan_text = tokenizer.decode(plan_ids[0], skip_special_tokens=True)

    # ── Phase 2: Generate action tokens (constrained to action vocab) ───────
    action_prompt = plan_text + f"\\nAction:"
    action_input  = tokenizer(action_prompt, return_tensors="pt").to(device)

    action_array = model.generate_action(
        pixel_values   = image.unsqueeze(0),
        use_constraint = True        # enforce action token masking
        **action_input
    )

    return {
        "plan":   plan_text,
        "action": action_array
    }

⚔️ RT-2 vs PaLM-E: Fundamental Design Differences

Both papers came from Google in 2023, both use giant VLMs for robotics, both were seminal. But they make fundamentally different architectural choices about where reasoning ends and control begins.

RT-2 is more elegant for learned manipulation skills. PaLM-E is more powerful for long-horizon symbolic planning. Modern systems like π0 (Physical Intelligence, 2024) combine both: a VLM backbone for planning and reasoning, plus a diffusion-based action expert for high-frequency, dexterous control — effectively getting the best of both worlds at the cost of complexity.

RT-2's influence on VLA design choices

Ideas RT-2 established

Action-as-language: treating actions as text tokens is now the dominant paradigm for discrete-output VLAs.

Co-fine-tuning: mixing robot + web data to prevent catastrophic forgetting is universally adopted.

VLA as term: coined by RT-2, now the standard name for this class of models.

Output constraints: logit masking at inference to enforce valid action tokens is standard practice.

Where the field moved beyond RT-2

Continuous actions: π0, OFT, and others use flow matching or diffusion for higher precision and 50 Hz control — discrete bins limit accuracy.

Multi-embodiment: Open X-Embodiment and OpenVLA showed multi-robot training is critical for generalization.

Action chunking: predicting action sequences rather than single steps improves temporal consistency.

Dual-system design: separating slow reasoning from fast control (Helix, Groot N1) for latency-dexterity tradeoff.

🎯 Key Takeaways for ML Engineers

1

Actions are just tokens

Discretize continuous motor commands into 256 bins, express as integers, train with cross-entropy. No new loss function, no new architecture. The insight is conceptual, not computational.

2

Co-fine-tuning is non-negotiable

Fine-tuning only on robot data causes catastrophic forgetting and collapses generalization (~38% vs ~62%). Keep web data in every batch.

3

Scale gives emergence

5B → 55B gives +15pp on emergent tasks. The capabilities RT-2 exhibits (symbol understanding, human recognition) only appear at sufficient scale.

4

LR from VLM pretraining

OpenVLA's ablation found 2e-5 (the original VLM pretraining LR) works best — not a lower LR as you'd expect for fine-tuning. The VLM weights need significant adaptation.

5

Chain-of-thought transfers

A technique invented for math LLMs (think before you answer) directly improves robot control quality. The mechanism is simple: prepend "Plan:" to the output format, fine-tune briefly.

6

Output constraint at inference

Add -1e9 to all non-action logits when in robot mode. This is required for reliability. Simple to implement, critical for safety.

7

256 bins is a precision limit

At ±50mm position range, each bin is ~0.4mm. Acceptable for pick-and-place, not for dexterous tasks. This is why π0 and OFT moved to continuous actions.

8

Web knowledge ≠ new motions

RT-2 can figure out to pick a rock (semantic reasoning). But if it was never trained to fold laundry, web knowledge alone won't teach it. Physical skills require robot demonstrations.

The 30-second summary for a colleague

RT-2's insight is delightfully simple: robot actions and words are both just sequences of discrete symbols — so why not use the same model for both? Discretize continuous motor commands into 256 bins, express them as integers, and fine-tune a 55B VLM to output them alongside regular language using the same cross-entropy loss. The result: a robot that figures out to pick up a rock when asked to hammer a nail — because the same weights that know about rocks and hammers from the web now also move the arm. The web's knowledge and the robot's muscles are no longer separated by an architectural wall.
